**Method:** 8 specialized review agents reading the post-pass-1 codebase in parallel
**Author:** Internal engineering review

This document is the source of truth for the **next** refactor session. Each finding has a severity, file:line location, problem statement, proposed fix, and risk note. All findings are **NEW** — items already in the original audit (or already applied/deferred per `REFACTOR-PASS-1.md`) have been excluded.

---

## How this audit was produced

After pass-1 landed (12 audit items applied, 163 tests passing, both Rust and frontend builds clean), 10 specialized review agents were dispatched in parallel. Each had a narrow scope, was given the original audit + the pass-1 summary as context, and was instructed not to re-flag known items. **Eight agents completed**; two (Build/Security + Testing Strategy) hit a rate limit and produced no findings — those scopes need to be re-run in the next session (see "Missing scopes" at the end).

## Top-of-mind issues across reviewers

Three patterns showed up in **multiple** agent reports independently and should be tackled first:

1. **Pause is purely cosmetic** (Architecture A1, Concurrency C1) — `pause_state` is set by `toggle_pause`, the tray menu, and `power_monitor`, and reported by `get_status`, but **no code reads it before recording or transcribing**. Toggling pause silences nothing. The audit's M1 fixed *how* pause is represented; this fixes *whether anyone listens*.
2. **`audio_engine` exports module-level `static`s that prevent test isolation and second-instance use** (Architecture A2, Concurrency C8, Audio AU1) — `pub static AUDIO_LEVEL` and `static SEGMENT_STARTED_AT` should become fields on `AudioEngine` plumbed through `AppState`.
3. **Filename-grammar coupling between `audio_engine`, `transcriber`, and `pipeline`** (Architecture A3/A4, Transcriber T2, Storage S8) — three modules independently parse `segment_YYYYMMDD_HHMMSS.wav`. A new `audio_dir` module that owns the filename grammar + cleanup primitives + orphan scan would unify the three sites.

Beyond those, the highest-impact NEW items are:

- **Real-time audio callback does disk I/O on the cpal RT thread** behind a coarse mutex with a nested resampler lock (Audio AU1, Concurrency C10) — the `rtrb` SPSC ring buffer is already a dependency but unused; the architecture is one refactor away from a lock-free handoff.
- **FTS5 update trigger is O(n²) per hour** (Storage S3) — every segment append re-tokenizes the whole growing hour text.
- **Append upsert is non-idempotent** (Storage S2) — any retry double-counts and duplicates text; needs a per-segment idempotency key.
- **`stderr(Stdio::inherit())` is a black hole in packaged macOS apps** (Transcriber T8) — pip install progress, VAD warnings, Python tracebacks all vanish; field debugging is blind.
- **Async `createEffect` blocks lose Solid tracking after the first `await`** (Frontend F1) — latent bug in `FilterPanel` waiting for the next reactivity edit.
- **`} catch (_) {}` everywhere swallows IPC errors** (Frontend F7, Types TY2) — fixable with a 20-line `tryInvoke` wrapper, no codegen needed.
- **Tailwind is imported but no utility classes are used** (CSS CS1) — full Tailwind preflight + ~10 KB of unused CSS shipping in the bundle.

# Pass-2 Review: Architecture & Module Cohesion

## Summary

Pass-1 successfully decomposed `lib.rs` into focused modules and the dependency graph is now acyclic (`commands`/`pipeline`/`tray` → `state`/`audio_engine`/`device_manager`/`storage`/`transcriber`/`events`). The remaining structural issues are: (1) `pause_state` is set in three places but **never read** by any module that records or transcribes — pause is currently UI theater, (2) `audio_engine` leaks two module-level `static` mutables (`AUDIO_LEVEL`, `SEGMENT_STARTED_AT`) that prevent more than one engine instance and tangle test isolation, and (3) several modules have shallow public surfaces that pass cleanup/event/timer responsibilities back to `pipeline.rs` instead of owning them.

## Findings

### A1. `pause_state` is write-only — recording and transcription continue while "paused" [severity: critical]

- Location: `src-tauri/src/audio_engine.rs:62-198` (callback never reads pause), `src-tauri/src/pipeline.rs:61-65` (loop never checks pause), `src-tauri/src/state.rs:24` (the field), set from `commands.rs:134`, `tray.rs:26`, `power_monitor.rs:30`.
- Problem: A grep for `pause` across `audio_engine.rs` and `pipeline.rs` returns zero hits. The audio callback writes WAV samples regardless of `PauseReason`, the segment-rotation timer fires regardless, the transcribe loop drains the channel regardless, and `process_orphans` runs at startup regardless. The audit's M1 fixed *how* pause is represented but not the fact that nobody reads it. `get_status` happily returns `is_recording: !is_paused` while the disk is being filled with audio. This is a correctness bug masquerading as an architecture issue: every other component depends on a pause signal that is purely cosmetic.
- Proposed fix: give `AppState` a single accessor `is_recording_allowed() -> bool` and consult it in three places: (a) the cpal callback early-return (drop samples, do not write, do not rotate); (b) `pipeline::start`'s `recv()` loop (skip transcription while paused, but keep draining); (c) `power_monitor`'s lock check (already gated). For (a), keep an `Arc<AtomicBool>` clone in `RecordingState` so the audio callback never takes a lock. The pause monitor flips this atomic whenever `pause_state` changes — wire it via a small `state::set_pause(reason)` helper instead of letting three call sites manipulate the `Mutex` directly.
- Risk: medium. Changes audible behavior. Add a test that `is_recording_allowed()` mirrors `pause_state.is_paused()` and a manual sanity check that toggling pause silences the WAVs.

### A2. `audio_engine` exports two module-level `static` mutables, blocking testability and second-instance use [severity: high]

- Location: `src-tauri/src/audio_engine.rs:36` (`pub static AUDIO_LEVEL: AtomicU32`), `src-tauri/src/audio_engine.rs:41` (`static SEGMENT_STARTED_AT: Mutex<Option<DateTime<Utc>>>`).
- Problem: Both are process-global. `AUDIO_LEVEL` is read by `commands::get_status` and `commands::subscribe_audio_level`; `SEGMENT_STARTED_AT` is read by `commands::get_status` via `current_segment_started_at()`. The `audio_engine` module name implies an instance you construct (`AudioEngine::new`), but the engine's *output* is communicated via globals. This is the textbook "shallow module" Ousterhout warns about: the `AudioEngine` handle hides almost nothing because the interesting state is reachable without it. Tests in `audio_engine.rs:378-405` already mutate `AUDIO_LEVEL` directly and would race if run concurrently. Spawning a second engine for any reason (device switch, A/B test, integration test) silently shares state.
- Proposed fix: make `AudioEngine` own this state. Add `level: Arc<AtomicU32>` and `segment_started_at: Arc<Mutex<Option<DateTime<Utc>>>>` as fields, expose `engine.audio_level()` and `engine.segment_started_at()`. Hand the `Arc`s to the cpal callback. Stash the `Arc<AudioEngine>` (or just the two `Arc`s) in `AppState` so `commands::get_status` reads through state instead of the global. Pipeline already constructs the engine and has the `Arc<AppState>` in scope — this is a one-edit re-routing.
- Risk: low–medium. Mechanical. Test isolation improves immediately.

### A3. `pipeline.rs` owns concerns that belong elsewhere; the module is a grab-bag [severity: high]

- Location: `src-tauri/src/pipeline.rs:14-20` (cleanup constants), `:97-131` (orphan scan), `:133-141` (orphan dedup), `:143-149` (cleanup timer thread).
- Problem: After the pass-1 split, `pipeline.rs` holds: pipeline orchestration, three audio-housekeeping constants, an orphan-WAV scanner that knows the `segment_*.wav` filename grammar, a date-parser that re-implements `extract_timestamp_from_path` from `transcriber.rs:161-167`, and a cleanup timer. The constants `MIN_SEGMENT_BYTES`, `MAX_RETAINED_SEGMENTS`, `AUDIO_MAX_AGE_SECS`, `CLEANUP_INTERVAL` describe *audio-dir lifecycle*, not pipeline orchestration. Meanwhile `audio_engine` exports the two cleanup primitives (`cleanup_old_segments`, `cleanup_old_audio`) but doesn't run them. The split is upside-down.
- Proposed fix: introduce `src-tauri/src/audio_dir.rs` (or rename `audio_engine`'s housekeeping fns into it) that owns: filename grammar (`SEGMENT_PREFIX`, `parse_segment_timestamp`), the cleanup constants, `cleanup_old_segments`, `cleanup_old_audio`, `find_orphan_segments(audio_dir) -> Vec<PathBuf>`, and `spawn_cleanup_timer(audio_dir)`. `pipeline.rs` shrinks to: spawn transcriber → call `audio_dir::find_orphan_segments` → spawn engine → spawn power monitor → spawn cleanup timer → drain `segment_rx`. `transcriber.rs::extract_timestamp_from_path` moves there too (it's used in both modules with the same parsing logic — see A4).
- Risk: low. Pure file move + re-export.

### A4. Filename-timestamp parsing is duplicated across `pipeline.rs` and `transcriber.rs` [severity: medium]

- Location: `src-tauri/src/pipeline.rs:133-141` (`already_transcribed`) and `src-tauri/src/transcriber.rs:161-167` (`extract_timestamp_from_path`).
- Problem: Both functions do the same thing — strip `segment_` prefix, parse `%Y%m%d_%H%M%S`, build a UTC datetime. Pipeline's version returns `bool` after going through a string `to_rfc3339()` round-trip; transcriber's returns `DateTime<Utc>` directly. If the filename grammar changes (e.g. add a device suffix) the orphan dedup silently drifts from the transcriber's view of "what time was this segment?". This is an Ousterhout "define errors out of existence" violation: the filename format is currently expressed implicitly in two regex-like strings instead of one.
- Proposed fix: single function in the new `audio_dir` module:
  ```rust
  pub const SEGMENT_PREFIX: &str = "segment_";
  pub const SEGMENT_TIME_FMT: &str = "%Y%m%d_%H%M%S";
  pub fn parse_segment_timestamp(path: &Path) -> Option<DateTime<Utc>> { ... }
  ```
  Both call sites consume this. `transcriber::extract_timestamp_from_path` becomes `parse_segment_timestamp(path).unwrap_or_else(|| Utc::now())`.
- Risk: none.

### A5. Toggle-pause logic is copy-pasted between `tray.rs` and `commands.rs` [severity: medium]

- Location: `src-tauri/src/tray.rs:24-31` and `src-tauri/src/commands.rs:133-141`.
- Problem: Identical `match *pause { None => Manual, _ => None }` in two places. If the toggle semantics change (e.g. promote system pauses to manual), it must change in two files. This is the same anti-pattern the audit's H2 flagged for transcribe-and-store.
- Proposed fix: add to `state.rs`:
  ```rust
  impl AppState {
      pub fn toggle_pause(&self) -> bool {
          let mut p = self.pause_state.lock();
          *p = if matches!(*p, PauseReason::None) { PauseReason::Manual } else { PauseReason::None };
          p.is_paused()
      }
  }
  ```
  Both call sites become `state.toggle_pause()`. Pairs naturally with A1's `set_pause` helper.
- Risk: none.

### A6. `pipeline::start` keeps `AudioEngine` and `PowerMonitor` alive via `_engine` / `_power_monitor` bindings — load-bearing underscore vars [severity: medium]

- Location: `src-tauri/src/pipeline.rs:44-57`.
- Problem: `let _engine = ...` and `let _power_monitor = ...` look like discarded values but are actually load-bearing — the `_engine`'s `Drop` would stop the cpal stream and the `_power_monitor`'s `Drop` would join the thread. Anyone who refactors the `recv()` loop to be non-blocking (or extracts it to its own function) will silently destroy audio capture. The convention `_x` means "intentionally unused" and that's wrong here. Also: there is no place to `Drop` these for graceful shutdown (see audit L8).
- Proposed fix: bind to real names (`let engine = ...; let power_monitor = ...`) and add a comment explaining lifetime. Better: stash both in `AppState` (or a new `AppState::pipeline_handles: OnceLock<PipelineHandles>`) so other modules (graceful shutdown, future restart-on-device-change) can reach them. This also makes A2's engine-state-in-AppState natural.
- Risk: low.

### A7. `state.rs` depends on `transcriber` only to hold a `Mutex<Option<Transcriber>>`, leaking init order across modules [severity: medium]

- Location: `src-tauri/src/state.rs:1-2` (`use crate::transcriber;`), `:23` (`pub transcriber: Mutex<Option<transcriber::Transcriber>>`), populated at `pipeline.rs:31` (`*state.transcriber.lock() = Some(t);`).
- Problem: `AppState::new` builds with `transcriber: None`, then `pipeline::start` later assigns `Some(t)`. Every consumer (`pipeline::transcribe_and_store:73-74`) must handle the `None` case ("Transcriber unavailable — segment dropped"). The lifecycle is "always present after pipeline init" but the type encodes "might never exist." This is a classic Ousterhout "define errors out of existence" miss: the `None` case is structurally impossible in production yet leaks `match` arms across the codebase. It also creates a circular-feel coupling: `state` knows about `transcriber`, `transcriber` is initialized by `pipeline`, `pipeline` initializes `state`'s field after the fact.
- Proposed fix: move `transcriber` ownership out of `AppState` into a `PipelineRuntime` owned by `pipeline.rs` (alongside `engine` and `power_monitor` from A6). `transcribe_and_store` becomes a method on `PipelineRuntime` and never sees `Option`. `AppState` shrinks to non-pipeline state (storage + pause + is_transcribing). Bonus: `state.rs`'s `use crate::transcriber` line goes away, and `state.rs` becomes a true leaf module.
- Risk: medium. Touches the `is_transcribing` flag's owner — keep it on `AppState` since `commands::get_status` reads it.

### A8. `audio_engine` is named for an engine but exports four free functions that don't need an engine [severity: medium]

- Location: `src-tauri/src/audio_engine.rs:26` (`segment_duration_secs`), `:43` (`current_segment_started_at`), `:296` (`cleanup_old_audio`), `:317` (`cleanup_old_segments`), plus `pub static AUDIO_LEVEL`.
- Problem: Five of the seven public items are not methods on `AudioEngine`. They're a config helper, two getters for module statics, and two filesystem cleanup utilities. The module is doing two unrelated jobs: (a) running a cpal stream, (b) being a junk drawer for "anything related to the audio dir." Ousterhout: a deep module hides a lot behind a small interface; this one exposes a lot of small unrelated entry points.
- Proposed fix: split (this is the natural follow-up to A3):
  - `audio_engine.rs`: only the `AudioEngine` struct + `new` + `Drop` + private callback machinery. `AUDIO_LEVEL` and `SEGMENT_STARTED_AT` become struct fields (per A2). `segment_duration_secs` becomes a private const or moves to a `config.rs`.
  - `audio_dir.rs`: cleanup + filename grammar + orphan scan (per A3 + A4).
- Risk: low. Mostly mechanical moves.

### A9. `subscribe_audio_level` polls a static every 33 ms from inside `commands.rs` — a coupled side door into audio state [severity: medium]

- Location: `src-tauri/src/commands.rs:121-130`.
- Problem: This Tauri command spawns a tokio task that wakes 30×/sec to read `audio_engine::AUDIO_LEVEL`. It's the only consumer of that polling pattern; everything else uses `Emitter::emit`. Every frontend call creates a new task that lives for the channel's lifetime — there's no cancellation if the frontend drops the channel for any reason other than send-error. The deeper issue: this is a `commands.rs` function that knows about audio_engine internals (the static) and has its own concurrency model that doesn't compose with anything else. The audit's M7 deferred fixing the polling, but the *placement* (a command that owns a 33ms loop) is itself a layering smell.
- Proposed fix: Even if you defer M7's "push instead of poll," move the polling task to `pipeline.rs` (or a new `pipeline::audio_level_publisher`) and have it `Emitter::emit` an event. The command then becomes a no-op or is deleted; frontend listens via `listen()`. This pulls the concurrency out of `commands.rs` (which should be thin request/response) and into the pipeline that already owns long-lived background work.
- Risk: low. Two-line frontend change to switch from `Channel` to `listen()`.

### A10. `events.rs` defines one constant, but two events are emitted from Rust — and one is the Tauri-internal channel name buried in `commands.rs` [severity: low]

- Location: `src-tauri/src/events.rs:4` declares `TRANSCRIPTION_UPDATED`. The `audio-level` channel in `commands.rs:122` is not registered as a named event but is the only other Rust→Frontend signal. Inside the codebase there's no `pub const` for the eventual `audio-level-changed` event from M7/A9. There are also no payload type aliases.
- Problem: `events.rs` is a one-liner today and will silently grow into a dumping ground unless given structure. It also doesn't include payload types, so `Emitter::emit(TRANSCRIPTION_UPDATED, &())` and the rumored future `("audio-level-changed", AudioLevelEvent)` have no shared schema. This is a shallow module right now: one constant doesn't justify a module.
- Proposed fix: either (a) inline the constant into `pipeline.rs` until there are 3+ events, or (b) flesh out `events.rs` with the missing constants and a `pub fn emit_transcription_updated(app: &AppHandle)` style helper that owns both the name *and* the payload type. The helper version is deeper (hides the `Emitter` import, the name string, and the payload all behind one fn) and is the right Ousterhout move once 2+ events exist.
- Risk: none.

### A11. `pipeline::start` blocks on `Transcriber::new()` before starting `AudioEngine`, so any Python startup latency = lost audio [severity: low]

- Location: `src-tauri/src/pipeline.rs:27-53`.
- Problem: The init order is (1) spawn MLX daemon and wait for `ready` handshake, (2) process orphans, (3) start audio engine. Steps 1+2 take seconds to tens of seconds (Python venv resolution, model load, orphan transcription). All audio during that window is dropped on the floor — the engine isn't capturing yet. For a "scribe" app whose value prop is "always recording," this is a meaningful gap.
- Proposed fix: start the audio engine first (it's cheap — just opens a cpal stream and starts buffering to disk). Then spawn the transcriber init on a background thread; once ready, drain whatever segments have already accumulated in the channel. Orphan processing should happen *after* the transcriber is up (it does today). The channel capacity is already unbounded so there's no backpressure concern.
- Risk: low. Re-orders three lines and moves transcriber init into a thread. Verify that `is_transcribing` and `get_status` handle the "engine running, transcriber not yet ready" intermediate state cleanly (today they would show `is_recording=true, is_transcribing=false` with a growing backlog, which is correct).
# Pass-2 Review: Concurrency, Threading & Lifecycle

## Summary
The post-refactor code is mostly sound, but several latent issues remain: the audio callback acquires multiple `parking_lot::Mutex` locks per buffer (including a global static and a resampler lock), the pipeline thread, power monitor, and cleanup timer have no shutdown handle so the process can hang on quit, the unbounded segment channel has no back-pressure, and most importantly **`PauseReason` is set but never read** by either the audio callback or the transcription loop — so toggling pause does nothing to either subsystem. None of these duplicate items already in the audit.

## Findings

### C1. `PauseReason` is set but never observed by audio/pipeline [severity: high]
- Location: `pipeline.rs:61-65`, `audio_engine.rs:137-184`, `state.rs:21-26`
- Problem: `pause_state` is updated by `commands::toggle_pause`, `tray.rs`, and `power_monitor.rs`, and reported back in `get_status`, but the audio callback continues writing WAVs and `pipeline::start` continues calling `transcribe_and_store` on every received segment. The system-pause-on-screen-lock path (the entire reason `power_monitor` exists) is therefore a no-op too. This is not a data race — it is missing wiring — but it is a lifecycle/coordination defect that fits this pass's scope.
- Proposed fix: in the `segment_rx.recv()` loop, check `state.pause_state.lock().is_paused()` before calling `transcribe_and_store`, and ideally also short-circuit `write_samples` (or at least `rotate_segment`'s `send`) when paused so disk and CPU stop. Cleanest separation: gate at the consumer (pipeline) since the audio engine is intentionally agnostic of pause.
- Risk: low — wires up behavior that the UI already advertises.

### C2. Audio callback grabs the global `SEGMENT_STARTED_AT` mutex on every rotation, contended with `get_status` [severity: low]
- Location: `audio_engine.rs:41-45, 252` and `commands.rs:85-87`
- Problem: `current_segment_started_at()` is called from `get_status` (a Tauri async command on a tokio worker) and the audio callback writes it inside `open_new_segment`. Both use `parking_lot::Mutex`, which is non-fair and uncontended-fast, so this is "merely inefficient locking" today. But `get_status` is polled at UI rate, and a `parking_lot::Mutex::lock()` inside an async fn blocks the async worker thread — if some future change makes the audio callback hold the lock longer (or if Tauri shrinks its blocking pool), the UI thread can be blocked by a real-time audio thread.
- Proposed fix: replace `Mutex<Option<DateTime<Utc>>>` with `AtomicI64` storing the epoch milliseconds (0 = unset). Lock-free read/write, no contention, and `get_status` doesn't block on a real-time thread.
- Risk: data race — none (atomic). Behavior — none.

### C3. Audio callback acquires resampler `Mutex` despite single-owner usage [severity: low]
- Location: `audio_engine.rs:87-107, 164-180`
- Problem: `resampler` is wrapped in `Arc<Mutex<SincFixedIn<f32>>>`, but it is only ever locked from inside the audio callback. The `Arc<Mutex>` is dead weight that costs a hash-cmpxchg per call — and worse, it makes the callback look as if it shares the resampler with another thread when it does not. This is "merely inefficient locking" but obscures actual ownership.
- Proposed fix: move the resampler into `RecordingState` (or capture it directly in the closure) and drop the `Arc<Mutex>`. The `state_clone.lock()` already provides exclusive access.
- Risk: none — refactor preserves behavior.

### C4. Pipeline thread, power monitor thread, and cleanup timer have no shutdown / `JoinHandle` [severity: medium]
- Location: `lib.rs:59-61` (pipeline), `pipeline.rs:143-149` (cleanup timer), `power_monitor.rs:9-13` (monitor)
- Problem: `lib.rs:59` does `std::thread::spawn(move || pipeline::start(...))` and discards the handle. `spawn_cleanup_timer` does the same. `PowerMonitor` keeps a `_handle` but never joins it and runs an unconditional `loop { sleep; ioreg }`. None of them ever exit. On app quit, Tauri tears down the runtime; these threads are killed mid-iteration. Concretely: the audio engine's `Drop` (if any) and the transcriber's child-process reaper (`transcriber.rs:117-122`) may not get to run — the cleanup timer thread's `cleanup_old_segments` could be holding a directory open, the pipeline loop could be mid-`transcribe()` blocking on `read_line`, and the spawned MLX Python child can be orphaned.
- Proposed fix: pair each long-lived thread with a `crossbeam_channel::Receiver<()>` shutdown signal selected against the work channel. Hold `JoinHandle`s in an `AppLifecycle` struct stored in `AppState`. Wire `app.on_window_event(|e| if matches!(e, CloseRequested) { lifecycle.shutdown() })` so quit drives orderly teardown of: stop audio stream → drain pending segments (or drop) → kill MLX child via `Drop` → join threads.
- Risk: medium — touches every long-running unit. Closely related to L8 in the audit but L8 was a one-line "future work" note; this is the concrete design.

### C5. Unbounded `crossbeam_channel` from audio engine to transcriber has no back-pressure [severity: medium]
- Location: `pipeline.rs:43`, `audio_engine.rs:291-293`
- Problem: `crossbeam_channel::unbounded::<PathBuf>()` is fine for `PathBuf` payloads (small), but if the transcriber stalls (slow first-segment, daemon hang per H9), the channel fills with paths. Each path corresponds to a 120 s WAV (~15 MB at 16 kHz f32 mono) sitting on disk. The cleanup timer at 30 min cadence and `MAX_RETAINED_SEGMENTS=6` will then race against a backlog: cleanup deletes old WAVs while their paths still sit in the channel, and the transcriber later receives a path to a file that no longer exists. Result: a flurry of "WAV open" errors when the transcriber catches up.
- Proposed fix: switch to `crossbeam_channel::bounded(MAX_RETAINED_SEGMENTS)`. On `send` failure (full), log and either drop the new segment or delete the oldest. Either way, keeps disk-state coherent with channel-state. Pair with H9 timeout work so a hung daemon doesn't fill the channel in the first place.
- Risk: medium — needs a decision on drop policy.

### C6. `transcriber.rs` re-entrancy: a Tauri command path that needs the transcriber would deadlock against the pipeline loop [severity: low / latent]
- Location: `transcriber.rs:42-72`, `pipeline.rs:69-92`
- Problem: `transcribe_and_store` holds `state.transcriber.lock()` for the *entire* `transcribe()` call (correctly — to serialize stdin/stdout). Today, no Tauri command touches `state.transcriber`. But the moment somebody adds e.g. a "transcribe this file" command (or a "restart MLX" command) and calls `state.transcriber.lock()` from a `#[tauri::command]`, that command will block the Tauri async worker for the entire duration of the in-flight segment (multiple seconds, sometimes tens of seconds). With enough such calls, all blocking workers are exhausted and the UI freezes.
- Proposed fix: add a doc-comment on `AppState.transcriber` warning that this lock can be held for >5s and must never be acquired from a `#[tauri::command]`. Alternative: route transcribe requests through the same `crossbeam_channel` worker so commands queue rather than block.
- Risk: livelock / queue starvation, not a deadlock. Pre-emptive note rather than current bug.

### C7. `Atomic Ordering::Relaxed` on `is_transcribing` is incorrect for "is_transcribing implies result has been stored" semantics [severity: low]
- Location: `pipeline.rs:70, 91`, `commands.rs:90`, `audio_engine.rs:36, 203, 209` (`AUDIO_LEVEL`)
- Problem:
  - `is_transcribing` — `Relaxed` is fine for a "UI hint" boolean that the reader doesn't use to make decisions about other memory. Acceptable today.
  - `AUDIO_LEVEL` — same, single `u32`, no ordering relationship to other memory. `Relaxed` is correct.
  - However, `is_transcribing.store(false)` happens *after* the storage write and the emit. If a future change makes the frontend call `get_timeline` immediately upon seeing `is_transcribing=false` and expects the new row to be visible, `Relaxed` would not provide the necessary happens-before with the SQLite write. That cross-store ordering needs `Release/Acquire`.
- Proposed fix: keep `Relaxed` for `AUDIO_LEVEL`. Switch the `is_transcribing.store(false, ...)` to `Release` and the matching `load` in `get_status` to `Acquire`, so any future code that uses the flag as a "results are visible" signal is correct. Cheap (no extra fence on x86/aarch64).
- Risk: behavior — none. Future-proofing.

### C8. `AUDIO_LEVEL` global static makes test ordering brittle and prevents multi-instance [severity: low]
- Location: `audio_engine.rs:36, 200-210`, also tests at `audio_engine.rs:378-406`
- Problem: `pub static AUDIO_LEVEL: AtomicU32` is process-global. The test `test_update_audio_level_attacks_fast_releases_slow` and `test_audio_level_ranges` mutate the same global; if Cargo runs tests in parallel (default), one test can flip `AUDIO_LEVEL` between another test's `store` and `load`. Functionally this is "merely flaky tests," not a production race.
- Proposed fix: move `AUDIO_LEVEL` into `AudioEngine` and expose via a getter held by `AppState` or a `Channel`-based push (matches audit M7). At minimum, mark the affected tests `#[serial]` (via the `serial_test` crate) or hold a `Mutex` around the test body.
- Risk: low.

### C9. `subscribe_audio_level` polls forever with no cancellation when client disconnects gracefully [severity: low]
- Location: `commands.rs:121-130`
- Problem: The loop only exits when `channel.send` returns `Err`. The frontend has no explicit way to cancel; the loop persists for the lifetime of the IPC channel. If many components subscribe and unsubscribe, this is a slow leak of tokio tasks (each holds 33ms timer + atomic load). Today only one subscriber exists, so it's "merely inefficient." But every navigation that re-mounts the status bar leaves the previous task spinning until Tauri tears down its channel — which can take minutes.
- Proposed fix: when M7 lands (event-push), delete this command. Until then, add a `tokio::select!` between `channel.closed()` (Tauri exposes this for Channel) and the sleep, so disconnect is observed within one tick.
- Risk: none.

### C10. Audio callback can be re-entrantly entered by cpal's restart-on-error path; `state` mutex would simply queue, but `parking_lot` does not detect this [severity: low / informational]
- Location: `audio_engine.rs:134-187`
- Problem: cpal's CoreAudio backend may invoke the data callback from a real-time thread; on stream errors it can also invoke the *error* callback (`audio_engine.rs:185`) on a different thread. The error callback only logs, so no contention today. However, `parking_lot::Mutex` is not re-entrant — if any future error-handling path calls back into code that locks `state` while the data callback is mid-`lock()`, a single-threaded re-entry would deadlock. Today's code does not do this; flagging because the audit refactor that consolidated state in a single `Arc<Mutex<RecordingState>>` makes this hazard easier to trip later.
- Proposed fix: keep error callback log-only; add a doc-comment on `RecordingState` that the lock must not be held across any callable that may itself lock `state`. Consider replacing the `Arc<Mutex<RecordingState>>` with a `crossbeam_channel` from the real-time thread to a writer thread (RT-audio best practice — never do filesystem I/O on the audio callback, which `write_sample` is).
- Risk: medium if the RT-thread/IO concern is taken seriously; low if just adding a comment. The current design does WAV writing on the cpal real-time thread, which can cause buffer underruns under disk pressure. Worth flagging separately from this pass.
# Pass-2 Review: Storage / SQL

## Summary
The pass-1 refactor improved correctness (atomic upsert, hour_slots-only orphan dedup, original-device preservation), but the storage module still has unresolved schema-evolution debt (legacy `transcriptions` table, no migration framework), several durability/perf pragmas left at SQLite defaults, FTS5 trigger amplification on long updates, two silently-swallowed-error sites, and an FTS sanitizer that allows operator-shaped tokens to leak. None of the integration tests in `tests/storage_*.rs` actually exercise `Storage` (they hit the legacy table via raw SQL), so the rowid contract change is not covered there but also not broken.

## Findings

### S1. Legacy `transcriptions` table created on every startup, never written, no migration plan [Medium]
- Location: `storage.rs:46-54`
- Problem: `Storage::new` still issues `CREATE TABLE IF NOT EXISTS transcriptions (...)` with seven columns. After C2 the code path that wrote to it was removed and the orphan-dedup query was switched to `hour_slots`. The table is now permanently dead schema. Worse, there is no `schema_version` table anywhere in the file, so the next breaking change has no safe path forward — anyone who installed before the C2 fix has data in `transcriptions` that will be invisible from the UI forever.
- Proposed fix: add an explicit version table and a one-shot migration. Keep `transcriptions` for a single release that copies surviving rows into `hour_slots`, then drop it.
  ```sql
  CREATE TABLE IF NOT EXISTS schema_version (version INTEGER PRIMARY KEY);

  -- v1 -> v2 migration (run inside a transaction)
  INSERT INTO hour_slots (hour_key, text, start_time, last_updated, device, segment_count)
  SELECT
    strftime('%Y-%m-%dT%H', start_time) AS hour_key,
    GROUP_CONCAT(text, ' '),
    MIN(start_time),
    MAX(start_time),
    COALESCE((SELECT device FROM transcriptions t2 WHERE t2.start_time = MIN(t1.start_time)), ''),
    COUNT(*)
  FROM transcriptions t1
  GROUP BY hour_key
  ON CONFLICT(hour_key) DO UPDATE SET
    text = hour_slots.text || ' ' || excluded.text,
    segment_count = hour_slots.segment_count + excluded.segment_count;

  DROP TABLE transcriptions;
  INSERT OR REPLACE INTO schema_version VALUES (2);
  ```
  Or, if the user base is known to be empty: just `DROP TABLE IF EXISTS transcriptions` and stop creating it.
- Risk: medium — migration on a populated DB needs a backup-and-rollback envelope. Document the version bump in REFACTOR-PASS-2.

### S2. Append upsert is non-idempotent — retries duplicate text and miscount segments [High]
- Location: `storage.rs:115-123`
- Problem: the upsert
  ```sql
  INSERT INTO hour_slots (...) VALUES (?1, ?2, ?3, ?3, ?4, 1)
  ON CONFLICT(hour_key) DO UPDATE SET
      text = text || ' ' || excluded.text,
      ...
      segment_count = segment_count + 1
  ```
  has no idempotency key. Pipeline calls this exactly once per successful transcription, but on any retry path (orphan-recovery double-fire, future at-least-once delivery, crash between `transcribe` and `append`) the same segment text is appended again and `segment_count` increments by 2. The `' '` separator is also blind to leading/trailing whitespace in either side: a `text` ending in ` ` plus a new chunk starting with ` ` produces three spaces; a chunk that is `""` produces a stray ` ` (today this is gated by `pipeline.rs` skipping empty results, but the contract isn't enforced by storage).
- Proposed fix: add a per-segment idempotency key and a single source of truth for the join character.
  1. Schema: `CREATE TABLE IF NOT EXISTS hour_slot_segments (slot_id INTEGER, segment_key TEXT PRIMARY KEY, ts TEXT NOT NULL)`.
  2. In `append_to_hour_slot`, take a `segment_key: &str` (e.g. the WAV stem `segment_20260416_143005`), `INSERT OR IGNORE` into `hour_slot_segments`, and only run the upsert when `changes() == 1`.
  3. Trim the inputs and use `printf` or a Rust-side joiner that collapses whitespace: `text = trim(text) || ' ' || trim(excluded.text)`. Reject empty `text` at the storage boundary with `Err(...)` rather than silently inserting `' '`.
- Risk: medium — adds a table + slight write cost. Eliminates a whole class of correctness bugs.

### S3. FTS5 update trigger reinserts the entire growing text on every append — O(n^2) per hour [High]
- Location: `storage.rs:73-77` (`hour_slots_au` trigger)
- Problem: every `append_to_hour_slot` is an UPDATE on `hour_slots`. The `hour_slots_au` trigger runs `'delete'` then `INSERT` of the **new** `text` value, which is the cumulative concatenated text for the whole hour. For an hour that grows to 60 segments * ~150 words = ~9000 words, FTS5 retokenizes the entire 9000-word document on segment 60, having already retokenized 8850 words on segment 59, and so on. Total work over the hour is quadratic in segment count.
- Proposed fix: switch FTS to per-segment rows (decoupled from `hour_slots`). With the segment table from S2:
  ```sql
  CREATE VIRTUAL TABLE hour_slot_segments_fts USING fts5(
      text, content='hour_slot_segments', content_rowid='rowid'
  );
  CREATE TRIGGER hssfts_ai AFTER INSERT ON hour_slot_segments BEGIN
      INSERT INTO hour_slot_segments_fts(rowid, text) VALUES (new.rowid, new.text);
  END;
  ```
  Search becomes a join `hour_slot_segments -> hour_slots`, with `GROUP BY slot_id` to dedupe matched slots. Each FTS write is O(segment), total per hour is O(n) words.
  If keeping the current schema, at minimum gate the trigger to skip when `new.text = old.text` (rare, but free).
- Risk: medium — schema change. Search ranking semantics change slightly (per-segment match vs whole-hour match); usually an improvement.

### S4. `sanitize_fts_query` allows the FTS5 column-filter operator to leak [Medium]
- Location: `storage.rs:229-247`
- Problem: the sanitizer keeps `'` and `-` and wraps each word in double quotes. Any single-character word survives — and FTS5 treats `*` inside a quoted prefix-search as literal, but a bare `-` token outside quotes is treated as the NOT operator. The filter `c.is_alphanumeric() || *c == '\'' || *c == '-'` permits `-foo` to survive as `"-foo"`. Quoted, FTS5 treats it as a literal phrase, so the operator path is mostly safe today. However:
  - The sanitizer does NOT escape internal double quotes inside the input — they're stripped, not escaped. Verified by the test `test_fts_sanitize` showing `test"broken` -> `"testbroken"` (loss of intent, not injection).
  - A query containing only punctuation (e.g. `:` or `*`) returns empty, which `search_hour_slots` already short-circuits — good.
  - But `c.is_alphanumeric()` is Unicode-aware and accepts characters from scripts that FTS5's default `unicode61` tokenizer treats as separators, producing tokens that match nothing — silent zero-result for users typing CJK, etc. Not an injection, but a usability cliff worth documenting (audit L6 already flagged the doc gap).
- Proposed fix: replace the char-filter with an explicit allowlist of FTS5-safe operators stripped + quote-escape, and document tokenizer behavior. Also add an FTS5 `tokenize='unicode61 remove_diacritics 2'` option for better recall.
  ```rust
  fn sanitize_fts_query(q: &str) -> String {
      q.split_whitespace()
          .map(|w| w.replace('"', "").replace('*', ""))
          .filter(|w| w.chars().any(char::is_alphanumeric))
          .map(|w| format!("\"{w}\""))
          .collect::<Vec<_>>()
          .join(" ")
  }
  ```
- Risk: low. Add a test for `"-foo"`, `"col:bar"`, and CJK input.

### S5. `query_map(...).filter_map(|r| r.ok())` swallows row-decode errors silently [Medium]
- Location: `storage.rs:138, 167, 188, 202`
- Problem: any `rusqlite::Error` from `map_hour_slot` (e.g. column type changed by a future migration, NULL where NOT NULL expected) is dropped. A timeline view that should display 50 rows might quietly render 47, with no log line and no user-visible error. Same pattern in `get_available_dates` (line 202).
- Proposed fix: collect into `Result<Vec<_>, _>`:
  ```rust
  let rows: Vec<HourSlot> = stmt
      .query_map(params![limit, offset], map_hour_slot)
      .map_err(|e| format!("Query failed: {e}"))?
      .collect::<rusqlite::Result<Vec<_>>>()
      .map_err(|e| format!("Row decode failed: {e}"))?;
  ```
  Three call sites; pure error-propagation change.
- Risk: low — surfaces previously hidden errors. Add a `log::warn!` if you want partial-result behavior.

### S6. Only `journal_mode=WAL` is set; the rest of the durability/perf pragmas are at default [Low/Medium]
- Location: `storage.rs:41`
- Problem: SQLite defaults are conservative for safety, costly for a single-writer append-mostly workload like this one. `synchronous=FULL` (default with WAL) does 2 fsyncs per commit; for a transcription-recovery app that already replays orphan WAVs on startup, `NORMAL` is the right trade. `temp_store`, `mmap_size`, and `cache_size` are also defaults.
- Proposed fix: bundle pragmas in one `execute_batch`:
  ```sql
  PRAGMA journal_mode=WAL;
  PRAGMA synchronous=NORMAL;
  PRAGMA temp_store=MEMORY;
  PRAGMA mmap_size=268435456;        -- 256 MB
  PRAGMA cache_size=-65536;          -- 64 MB page cache (negative = KB)
  PRAGMA wal_autocheckpoint=1000;
  PRAGMA foreign_keys=ON;            -- in case S2's segment table adds an FK
  ```
- Risk: low. `synchronous=NORMAL` can lose the last few seconds of writes on OS crash (not power loss) — acceptable here because the WAVs on disk are the source of truth for orphan recovery.

### S7. `conn.prepare(...)` per call instead of `prepare_cached` [Low]
- Location: `storage.rs:133, 162, 183, 195`
- Problem: every read recompiles the statement. With a single mutex and short statements the cost is small, but it's free to fix and removes a recurring CPU burn during search-as-you-type and timeline scrolling.
- Proposed fix: `conn.prepare_cached(&sql)`. Note the SQL strings built via `format!` should be `const`-folded where possible — pull the column list out as a `&'static str` and use a `&'static str` query string when there's no interpolation (`get_available_dates` qualifies today; `get_hour_slots`/`search_hour_slots`/`get_slots_by_date_range` can all be `&'static str` if you stop `format!`-injecting `HOUR_SLOT_COLUMNS` and just inline it).
- Risk: none.

### S8. `start_time` exact-match dedup is brittle to TZ drift between writer and orphan-replay [Medium]
- Location: `storage.rs:211-220`, called from `pipeline.rs:140`
- Problem: the writer stores `timestamp.to_rfc3339()` of a `DateTime<Utc>` derived from the audio engine's `SEGMENT_STARTED_AT`. The orphan replay path in `pipeline.rs:133-141` parses the WAV stem (e.g. `segment_20260416_143005`) as `NaiveDateTime`, then calls `.and_utc().to_rfc3339()`. If the stem timestamps were written by the audio engine in **local** time (verify in `audio_engine.rs::open_new_segment`) but interpreted as **UTC** here, every orphan misses dedup whenever the machine's local TZ != UTC, and we re-transcribe + re-append on every restart — exactly the pre-C2 bug, just hidden behind a TZ assumption. The audit's note "this match is therefore weak" hints at it; pass-1 didn't fix it.
- Proposed fix: store an explicit `segment_id` (the WAV stem) per-segment in the S2 segments table, and dedup on that string instead of an RFC3339 round-trip. Until then, audit `audio_engine.rs::open_new_segment` to confirm it formats the stem in the same TZ that `to_rfc3339` would emit (UTC), and add a test that simulates a non-UTC local TZ.
- Risk: medium — requires touching audio_engine to expose the stem.

### S9. No transaction wrapper around the orphan-replay batch [Low]
- Location: callers in `pipeline.rs:118-128` (loop calling `transcribe_and_store` per orphan)
- Problem: each orphan triggers an independent `INSERT ... ON CONFLICT` with its own implicit `BEGIN/COMMIT`. For N orphans (after a long offline stretch this can be dozens), N fsyncs at WAL `synchronous=FULL`. Wrapping the whole orphan loop in a single `BEGIN IMMEDIATE; ... COMMIT;` collapses to one fsync. Storage doesn't currently expose `transaction()`, so the fix has two parts.
- Proposed fix: add `pub fn with_transaction<F: FnOnce(&Connection) -> Result<()>>(&self, f: F)` on `Storage` (or a thinner `pub fn append_batch(&self, items: &[(text, ts, device)])`), then in `pipeline.rs::process_orphans` collect transcribed results and write them in one transaction. If S6 lands and `synchronous=NORMAL`, the urgency drops — but batch is still cheaper.
- Risk: low.

### S10. `hour_key` lexical ordering is correct only because of zero-padding; not enforced by schema [Low/Nit]
- Location: `storage.rs:105-111` (writer) and `storage.rs:179-181` (`get_slots_by_date_range`)
- Problem: queries like `WHERE hour_key >= ?1 AND hour_key <= ?2` and `ORDER BY hour_key ASC` only work because the writer happens to format `{:04}-{:02}-{:02}T{:02}`. If a future caller ever inserts an unpadded key (e.g. `2026-4-6T9`), range queries silently break and FTS results scramble. There's no `CHECK` constraint and no UNIQUE-by-shape enforcement beyond `UNIQUE(hour_key)`.
- Proposed fix:
  ```sql
  ALTER TABLE hour_slots ADD CONSTRAINT hour_key_shape
      CHECK (length(hour_key) = 13 AND substr(hour_key, 11, 1) = 'T');
  ```
  SQLite doesn't support `ADD CONSTRAINT` post-hoc, so this has to land at table-creation time inside the migration. Alternative: add a Rust-side `fn format_hour_key(local: DateTime<Local>) -> String` used by both writer and any future tooling, plus a debug-assert.
- Risk: none.

- M2/M4 (row mapper + ON CONFLICT): fixed in pass-1.

# Pass-2 Review: Audio Engine

## Summary

Pass-1 cleaned up the surface (logging, smoother constants, error paths, single source of truth for segment start), but the audio callback hot path is still allocation-heavy and holds a coarse mutex around the entire write — including a nested `Mutex<SincFixedIn>` lock and synchronous `BufWriter<File>` I/O. The real-time thread can therefore block on disk I/O and on a contended `parking_lot::Mutex` while two `Vec` allocations and one `VecDeque::drain().collect()` run per chunk. `rtrb` is already a dependency but unused; the architecture is one refactor away from a lock-free SPSC handoff. Several smaller correctness/UX items (no graceful stream stop, no device hot-swap, dead `samples_written`, dual segment-start sources, two directory walks for cleanup, 32-bit float WAV doubling disk I/O vs. int16) round out this pass.

## Findings

### AU1. Real-time callback holds a single mutex over disk I/O AND a nested resampler lock [HIGH]
- Location: `audio_engine.rs:162-183` (the `state_clone.lock()` covers everything from resample-buffer push through `write_samples()`, which calls `writer.write_sample()` on a `BufWriter<File>`); the inner `resampler.lock()` at `:169` is taken *while still holding* the outer state lock.
- Problem: the cpal callback is the audio engine's only real-time thread. While that lock is held, no other thread can read `RecordingState` (e.g. for stats, graceful rotation, or shutdown), and the lock-hold duration includes (a) the resampler's sinc-convolution work — which at `sinc_len=256, oversampling_factor=256` is non-trivial — and (b) writes to a `BufWriter<File>` that can block whenever the buffer flushes (~8 KiB default; a 16 kHz f32 segment fills it ~488×/segment). On a stalled fs (Spotlight reindex, Time Machine snapshot, sleeping disk) the audio thread blocks and cpal will start dropping frames or under-running. The nested `resampler.lock()` is also pointless: only one thread (the callback) ever uses the resampler — it does not need a `Mutex` at all; a `RefCell` would suffice if the callback's `Fn` requirement weren't blocking that. The right move is to keep resampler state owned by the callback closure (move it in once, no `Arc<Mutex<…>>`).
- Proposed fix: introduce an `rtrb::RingBuffer<f32>` (already a dep — `Cargo.toml:27`) sized to ~2 s of mono 16 kHz audio (~32 K floats, 128 KiB). Callback path = downmix → resample → push to ring → return. A dedicated **writer thread** drains the ring into the WAV writer and handles rotation/cleanup. Move the `SincFixedIn` into the callback closure (no `Arc<Mutex>`); move `WavWriter` and `RecordingState` into the writer thread. The shared state across threads becomes only the `segment_tx` and a tiny `AtomicU64` for `samples_written` (or drop it — see AU8).
- Risk: medium. Concurrency restructure, but `rtrb` is SPSC and the ownership split is clean. Keep behavior parity for segment rotation by signaling the writer thread via a control channel. Existing unit tests on `update_audio_level`, downmix, and resample-buffer accumulation continue to pass since they don't depend on the mutex topology.

### AU2. Per-callback heap allocations on the hot path [HIGH]
- Location: `audio_engine.rs:152-158` (`mono: Vec<f32>` allocated *every callback*, even in the mono passthrough branch via `data.to_vec()`); `:168` (`let chunk: Vec<f32> = s.resample_buf.drain(..RESAMPLE_CHUNK).collect()` allocates a fresh 1024-float `Vec` per resample chunk — at 48 kHz native rate that's roughly 47×/segment-second).
- Problem: at 48 kHz native, cpal typically delivers callbacks ~85–200×/sec (buffer 256–512 frames). Each callback path allocates: (a) one `Vec<f32>` of length `data.len() / native_ch`, plus (b) one `Vec<f32>` of 1024 floats per resample chunk drained. That is 100s of allocations/sec on a real-time thread — exactly the pattern that triggers the macOS scheduler to demote audio threads. Quantification: at 48 kHz / 480-frame callbacks, ~100 callbacks/sec × (1 mono Vec + ~1 chunk Vec on average) = ~200 allocations/sec, ~48 KiB/sec allocator pressure. Trivial in absolute terms, but allocator latency spikes are the failure mode, not throughput.
- Proposed fix: pre-allocate two buffers owned by the callback closure (`mono_buf: Vec<f32>` and `chunk_buf: Vec<f32>` with capacity `RESAMPLE_CHUNK`). Reuse via `mono_buf.clear(); mono_buf.extend(downmix_iter)`. Replace `s.resample_buf.drain(..RESAMPLE_CHUNK).collect()` with `chunk_buf.clear(); chunk_buf.extend(s.resample_buf.drain(..RESAMPLE_CHUNK))`. Even better, replace `VecDeque` with a fixed-size SPSC ring (AU1) and a contiguous `&[f32]` slice handoff to the resampler.
- Risk: low. Pure allocation hygiene; semantics unchanged.

### AU3. WAV format choice: 32-bit float doubles disk I/O for no benefit [MEDIUM]
- Location: `audio_engine.rs:241-242` (`bits_per_sample: 32, sample_format: SampleFormat::Float`).
- Problem: a 2-minute segment at 16 kHz f32 = 7.68 MB; in int16 PCM it would be 3.84 MB. mlx-whisper / whisper.cpp converts to f32 internally regardless, so the on-disk format is purely a transport choice. With `WHISPER_SEGMENT_SECS=120` (default) that's 230 MB/hr vs. 115 MB/hr — directly affects the BufWriter flush cadence (AU1) and the cleanup heuristic. Float WAV is also slightly more brittle (some readers don't support WAVE_FORMAT_IEEE_FLOAT). At a calibrated input gain (clipping unlikely on a quiet speech mic) the SNR loss from int16 quantization is far below the noise floor of any consumer mic.
- Proposed fix: switch to `bits_per_sample: 16, sample_format: SampleFormat::Int` and convert with `(s.clamp(-1.0, 1.0) * 32767.0) as i16` in the writer. Halves disk I/O and BufWriter flush frequency. Verify the Python transcriber side handles int16 WAV (mlx-whisper does; it loads via `audioread`/`librosa`). Keep f32 as an env-var opt-in if anyone wants archival-quality.
- Risk: low. Audible quality change is imperceptible for ASR. One-line WAV-spec change + a sample-conversion function.

### AU4. Resampler quality settings are GPU-rendering-grade for ASR-grade audio [MEDIUM]
- Location: `audio_engine.rs:88-94` (`sinc_len: 256, oversampling_factor: 256, BlackmanHarris2, Linear`).
- Problem: `sinc_len * oversampling_factor` = 65,536-tap effective filter bank. This is overkill for a 16 kHz mono ASR pipeline where whisper's mel-spectrogram throws away anything above 8 kHz anyway. Per-callback CPU cost in the resampler is (rough) `sinc_len * input_chunk` multiplies = `256 * 1024 ≈ 262K` MACs per `RESAMPLE_CHUNK`. At 48 kHz that runs ~47×/sec → ~12M MACs/sec just for resampling, on the audio thread. Audible quality difference vs. `sinc_len=64, oversampling_factor=128` for speech: zero.
- Proposed fix: drop to `sinc_len: 64, oversampling_factor: 128` (or even `FastFixedIn` / `FftFixedIn`). Profile both before merging — `cargo bench` or `criterion` against a 60 s 48 kHz buffer. Expected ~10–20× speedup with no transcription-accuracy delta. This change also reduces lock-hold duration in AU1.
- Risk: low. Quality change is below ASR's noise floor; trivial revert if a corpus A/B shows regression.

### AU5. Two sources of truth for segment start time [MEDIUM]
- Location: `audio_engine.rs:54` (`segment_started: std::time::Instant` on `RecordingState`) and `:41` (`SEGMENT_STARTED_AT: Mutex<Option<DateTime<Utc>>>` static).
- Problem: pass-1 added the `SEGMENT_STARTED_AT` static to give `get_status` a wall-clock view (REFACTOR-PASS-1.md, M9), but the original `segment_started: Instant` was kept for the rotation check at `:228`. Now the same concept ("when did the current segment start?") lives in two places, set in two places (`open_new_segment` updates both at `:249, :252`), and the `Instant` and `DateTime` can drift if the system clock changes mid-segment (unlikely but possible across NTP step or sleep/wake). More importantly, it's the kind of "two fields that must always agree" that breaks during the next refactor.
- Proposed fix: keep only `SEGMENT_STARTED_AT: DateTime<Utc>` and derive elapsed via `(Utc::now() - started).num_seconds()`. Wall-clock arithmetic is fine here because the rotation tolerance is seconds, not microseconds; it also makes "wall-clock vs. monotonic" explicit. Alternative: keep only `Instant` on the writer thread (AU1) and expose a derived `DateTime` via an `AtomicI64` (millis-since-epoch) read by `get_status`.
- Risk: low. Behavior identical unless clock steps mid-segment, in which case the new behavior is arguably more correct (UI agrees with file timestamps).

### AU6. `cleanup_old_segments` and `cleanup_old_audio` walk the same directory twice [LOW]
- Location: `audio_engine.rs:296-315` (`cleanup_old_audio` — by mtime age) and `:317-336` (`cleanup_old_segments` — by count). Both `read_dir` the audio_dir and filter by `.wav` extension.
- Problem: with M8 in place, the cleanup thread runs both functions back-to-back every 30 minutes, causing two stat-storms over the same directory. Trivial today (~30 files) but the functions also have inconsistent error handling — `cleanup_old_audio` swallows the `remove_file` error, `cleanup_old_segments` logs it.
- Proposed fix: merge into one `cleanup_audio_dir(audio_dir, max_age_secs, max_segments)` that walks once, collects `(path, mtime)`, applies both rules, and dedupes the deletion set. Consistent error handling.
- Risk: none. Pure consolidation.

### AU7. No graceful shutdown path on the cpal stream [MEDIUM]
- Location: `audio_engine.rs:47-49` (`AudioEngine` only holds `_stream: cpal::Stream`; relies entirely on `Drop`).
- Problem: when the app is quit, `Drop` on `cpal::Stream` calls `pause` and tears down the device. But (a) the in-flight callback may be mid-write to the WAV file, (b) `RecordingState.writer` is *not* finalized — `WavWriter::finalize()` is never called, leaving the last segment with an unfinished RIFF header (incorrect data size). The orphan-recovery path on next launch will see a truncated WAV; whether `hound` rejects it depends on platform. There is also no signal to the writer thread (once AU1 lands) to drain and finalize.
- Proposed fix: implement `impl Drop for AudioEngine` that (i) `stream.pause()`, (ii) sends a `Shutdown` control message to the writer thread (AU1), (iii) blocks briefly (≤500 ms) for the writer thread to finalize the current WAV. Also expose a `pub fn shutdown(self)` that consumes self for explicit async-friendly teardown from `pipeline.rs`.
- Risk: low. Adds a finite shutdown wait but eliminates a class of corrupted-segment bugs.

### AU8. `samples_written: u64` is dead state [LOW]
- Location: `audio_engine.rs:53` (field), `:225` (incremented), `:263, :270-273` (used only in one log line on rotate).
- Problem: this counter is computed on every `write_samples` call (one add per chunk) but its only consumer is a log message. The same number is recoverable from `WavWriter::duration()` or from the file size after finalize. Not a hot-path cost (one add per chunk) but it adds a field that the ring-buffer refactor (AU1) would otherwise not need to share between threads.
- Proposed fix: drop the field. If the log line is valuable, compute samples-written from `writer.duration()` or `samples_written = writer.len() as u64 / 4` (post-finalize, from file size). Or read from the WAV header on the consumer side.
- Risk: none. Removing dead state.

### AU9. Smoother math: `(u32 → f32 → u32)` round-trip can underflow `release` to zero [LOW]
- Location: `audio_engine.rs:200-209` (`update_audio_level`).
- Problem: `(prev as f32 * 0.92 + new_level as f32 * 0.08) as u32` truncates toward zero. When `new_level == 0` and `prev == 1`, the result is `0.92 as u32 == 0` — i.e. the smoother reaches zero in one tick from level 1, not asymptotically. Same for `prev == 2 → 1.84 → 1`. Combined with the test at `:405` ("release should be slow, not instant") this means the test passes in the 0→100→0 case but fails the spirit at low levels. There's also a subtle bias: `as u32` truncates instead of rounds, so the smoother systematically under-reports compared to `.round() as u32`.
- Proposed fix: use `.round()` and saturating arithmetic. Or store `AUDIO_LEVEL` as a fixed-point u32 (e.g. value × 256) so the smoother has sub-integer resolution. Simplest patch: `((prev as f32 * 0.92 + new_level as f32 * 0.08).round() as u32).min(LEVEL_MAX as u32)`.
- Risk: none. Cosmetic UI improvement at low audio levels.

### AU10. No device-disconnect recovery and no UI device picker [MEDIUM]
- Location: `audio_engine.rs:64-67` (host + device picked once at `AudioEngine::new`), `:185` (stream error callback only logs).
- Problem: the cpal `error_callback` at `:185` logs the error and does nothing. If the user unplugs the USB mic, switches output to AirPods (which on macOS sometimes recreates the input device), or the OS shifts default input, the stream goes dead silently — recording stops, no segment rotation, `AUDIO_LEVEL` freezes at its last value, the UI shows green-light status and the user thinks everything is fine. There's also no way to choose a non-default device from the UI; `device_manager::select_best_input` always wins.
- Proposed fix (two parts):
  1. **Hot-swap**: on `error_callback`, signal a supervisor thread that tears down the stream and rebuilds via `device_manager::select_best_input` with backoff. Emit `audio-device-changed` to the frontend so the user sees it.
  2. **Picker**: add `commands::list_input_devices() -> Vec<DeviceInfo>` and `set_input_device(id: String)`; persist choice in storage. `device_manager::list_input_devices` already exists (audit M23) — wire it.
- Risk: medium. Hot-swap interacts with AU7 (graceful shutdown of the old stream) and AU1 (writer thread state). Worth it: silent-failure on device disconnect is the #1 "I lost my recording" footgun in apps like this.

# Pass-2 Review: Transcriber & Python Daemon

## Summary
The IPC layer works but the Rust↔Python contract is implicit, undocumented, and unversioned. Several brittle couplings (filename-encoded timestamps, hardcoded interpreter paths, six candidate script paths) and unbounded I/O (`read_line` with no cap, inherited stderr, no WAV pre-validation) make the daemon fragile in a packaged macOS build. Findings below are scoped to genuinely new issues not already in the audit.

## Findings

### T1. IPC protocol has no version, no schema, and no handshake fields beyond status/model/vad [HIGH]
- Location: `src-tauri/scripts/mlx_transcribe.py:211` (ready frame); `src-tauri/src/transcriber.rs:142-150` (ready parser)
- Problem: The handshake is `{"status":"ready","model":..., "vad":...}` — no `protocol_version`, no schema id, no list of supported request fields. If the request shape changes (e.g. adding `language`, `prompt`, `priority`), Rust and Python can drift silently and a stale daemon (cached venv, old install) will silently misbehave with no detection. There is also no per-request `id` for correlation, so any future move to pipelined or async requests is impossible without a breaking change.
- Proposed fix: Bump to a typed protocol: ready frame includes `protocol_version: 1`, `capabilities: ["vad","silence_gate","language_en"]`, `request_schema: {...}`. Rust hard-fails on version mismatch (rather than producing wrong transcriptions). Add an optional monotonically-increasing `id` echoed in responses.
- Risk: Low — additive on both sides. Worth doing before the protocol gains more fields.

### T2. `extract_timestamp_from_path` couples transcriber to audio_engine's filename format [HIGH]
- Location: `src-tauri/src/transcriber.rs:161-167`; producer is `audio_engine.rs::open_new_segment` (filename `segment_YYYYMMDD_HHMMSS.wav`)
- Problem: Two unrelated modules agree on a string format via convention only. Change the format in `audio_engine` (e.g. ISO-8601, microsecond precision, device tag) and `transcriber` silently falls back to `Utc::now()` — which is wrong on the orphan-recovery path (segments may be hours old). No compile-time check, no test that the producer/consumer agree.
- Proposed fix: Introduce a typed `SegmentRef { path: PathBuf, started_at: DateTime<Utc>, expected_duration: Duration }` carried over the existing crossbeam channel. `extract_timestamp_from_path` becomes a fallback used only by orphan recovery, and the orphan path gets its own explicit parser with a documented format constant shared from `audio_engine`.
- Risk: Low — refactor is mechanical; channel payload widens.

### T3. `count_samples` re-opens the WAV solely for a duration that the producer already knows [MEDIUM]
- Location: `src-tauri/src/transcriber.rs:35-37, 156-159`
- Problem: Every transcription does an extra `WavReader::open` + header parse just to compute `end_time`, which (a) adds I/O on the hot path, (b) is the only thing the WAV file is read for in Rust, and (c) `end_time` is never actually consumed by any caller — it's a dead field flagged by the audit. If kept, it should ride on the segment message. If not, both `count_samples` and `end_time` should go.
- Proposed fix: Pass `samples_written: u64` (or `duration: Duration`) on the segment channel from `audio_engine`. Delete `count_samples` and the `end_time` field entirely (per dead-code warning).
- Risk: None — purely subtractive after T2 lands.

### T4. `find_python` is locale-fragile and silent about which interpreter was selected [MEDIUM]
- Location: `src-tauri/src/transcriber.rs:169-176`
- Problem: Hardcoded list `/opt/homebrew/bin/python3.14`, `/opt/homebrew/bin/python3`, `/usr/local/bin/python3`, `python3`. On Apple Silicon without Homebrew, on Intel Macs, on systems with pyenv/asdf/uv, the hit pattern is unpredictable. Worse: it accepts ANY interpreter that responds to `--version` — including Python 3.8 — and `mlx-whisper` requires 3.10+. Then `pip install --break-system-packages` runs against whatever it found, possibly polluting the user's system Python. Currently no log line records which interpreter was chosen, so support debugging is blind.
- Proposed fix: (a) Require `>=3.10` via `Command::new(p).args(["-c", "import sys; sys.exit(0 if sys.version_info >= (3,10) else 1)"])`. (b) Log the resolved path + version at INFO. (c) Honor an env var `WSCRIBE_PYTHON` override. (d) Combine with audited M11 (per-app venv) so the system Python is never mutated.
- Risk: Low.

### T5. `find_script` has six candidate paths with no tie-breaking and no log of which won [MEDIUM]
- Location: `src-tauri/src/transcriber.rs:178-200`
- Problem: Six relative candidates probed in order. In a dev tree where multiple checkouts exist, or where `cargo run` is invoked from a parent dir, the wrong script can be picked. The code silently uses the first match without logging which one or its mtime. If a stale copy exists in `Resources/`, the bundle ships with one version and dev runs another.
- Proposed fix: Single source of truth — embed the script via `include_str!("../scripts/mlx_transcribe.py")` and write it to `$APP_DATA_DIR/mlx_transcribe.py` on startup with a content hash for idempotency. Eliminates the search entirely, guarantees dev/bundle parity, and makes the script self-contained in the binary.
- Risk: Low — the script is ~200 lines, well under any reasonable embed budget.

### T6. `device` is queried at response time, not capture time — can disagree with the actual recording device [MEDIUM]
- Location: `src-tauri/src/transcriber.rs:94, 104` (`device_manager::get_current_device_name()` called after the daemon returns)
- Problem: A segment recorded 30s ago on AirPods is labelled with whatever device is currently selected when the transcription completes. If the user switched to MacBook Speakers mid-segment or between segments, the stored `device` is wrong. The `audio_engine` is the only module that knows which device actually captured the bytes.
- Proposed fix: Carry `device: String` alongside the segment path in the channel payload (T2). The transcriber stops calling `device_manager` entirely; it just forwards the truth.
- Risk: Low — the audit's M5 already touches device semantics; this is the producer-side complement.

### T7. `read_line` is unbounded — a runaway daemon can OOM the host [MEDIUM]
- Location: `src-tauri/src/transcriber.rs:67-72, 142-143`
- Problem: `BufReader::read_line` reads until newline with no size cap. A buggy daemon (or a malicious WAV pointer that triggers a Python error containing massive stack output redirected to stdout — see T8) can grow the line buffer until the Tauri process OOMs. For a transcription line the ceiling should be ~256 KB at the absolute most.
- Proposed fix: Replace with `take(MAX_LINE_BYTES).read_until(b'\n', &mut buf)`. On overflow, kill the daemon and surface a clean error.
- Risk: None — bounded read is strictly safer.

### T8. `stderr(Stdio::inherit())` is a black hole in a packaged macOS app and can corrupt stdout framing [HIGH]
- Location: `src-tauri/src/transcriber.rs:134`; Python writes diagnostics to stderr at `mlx_transcribe.py:17, 35`
- Problem: In a `.app` bundle launched from Finder, the host's stderr goes nowhere — no Console.app, no log file, gone. Pip install progress, VAD-unavailable warnings, mlx-whisper download progress, and any Python tracebacks vanish, making field debugging impossible. Additionally: `mlx_whisper` and `huggingface_hub` are both known to write progress bars to stdout in some configurations — if that ever happens, it corrupts the JSON-line frame and crashes the parser (T7 makes that worse).
- Proposed fix: `Stdio::piped()` for stderr; spawn a background thread that line-reads it and forwards to `log::warn!("[mlx-py] {line}")`. Also explicitly set env `TQDM_DISABLE=1`, `HF_HUB_DISABLE_PROGRESS_BARS=1`, `PYTHONUNBUFFERED=1` on spawn to guarantee stdout stays JSON-only.
- Risk: Low.

### T9. Daemon spawn happens on the pipeline thread, blocking first-segment latency by ~3-8s of model load [MEDIUM]
- Location: `src-tauri/src/transcriber.rs:26-31` called synchronously from `pipeline::start_pipeline` setup
- Problem: `Transcriber::new` returns only after the Python process has spawned, imported `mlx_whisper`, loaded Silero VAD, and emitted the ready frame. On a cold start with the model already cached this is several seconds; on first-ever launch it includes the model download (multi-GB). The pipeline thread can't accept the first segment until that completes — so the very first 30s recording sits in the channel buffer waiting.
- Proposed fix: Spawn daemon on a dedicated thread in parallel with `AudioEngine::new()`. Use a `OnceCell<Arc<Transcriber>>` that the pipeline thread `.wait()`s on only when it actually has a segment to process. Bonus: emit a `daemon-warming` event so the UI can show progress on first launch.
- Risk: Low.

### T10. No WAV validation before send, no retry/backoff for transient daemon failures, daemon `pip install` happens after `ensure_package` import attempt — race + restart-loop hazard [LOW]
- Location: `src-tauri/src/transcriber.rs:33-72`; `scripts/mlx_transcribe.py:13-23, 218-230`
- Problem: Three small issues, all related to fault handling:
  1. Rust sends any path string the channel hands it. A truncated WAV (write_sample failed mid-segment, finalize partial) is sent to Python, which raises an exception — Rust then kills + respawns the daemon at full cost. A 100-byte `is_well_formed_wav(path)` check (header magic + RIFF size sanity) avoids the round-trip.
  2. On daemon write/read failure, Rust drops the daemon and returns Err immediately; the next call respawns. There's no backoff — if the model is genuinely broken (corrupt cache, OOM), this respawns the daemon on every segment, ~once per 30s, forever. Add an exponential backoff with a circuit-breaker that pauses transcription and surfaces to the UI after N consecutive failures.
  3. Python `ensure_package` does the `__import__` at module load time, so the install only runs on `ImportError`. But pip's stdout is `--quiet` and goes to stderr (T8 black hole); if install fails (network down, disk full), the next `import mlx_whisper` raises and the daemon dies on stdin EOF before sending a ready frame, so Rust's `read_line` returns "" and reports "Daemon ready failed" with no actionable detail. Capture pip's exit code and emit a structured error frame instead of dying.
- Proposed fix: As described — pre-validate WAVs, add backoff + circuit-breaker, emit structured ready-failure frames.
- Risk: Low.
# Pass-2 Review: Frontend State & Effects

## Summary
The post-pass-1 frontend leaves several reactivity, lifecycle, and resource-leak hazards untouched that lie outside the audit's already-flagged structural items (M17/M18/M19/M22, L11/L12/L13). Most are subtle: async effects that lose Solid tracking after `await`, timers/listeners that race or leak across unmounts, and module-scope mutable state that survives in dev/HMR. None are critical, but several are easy to fix and reduce visible bugs.

## Findings

### F1. Async `createEffect` blocks lose reactive tracking after the first `await` [HIGH]
- Location: `src/components/FilterPanel.tsx:21-29` and `src/components/FilterPanel.tsx:31-47`.
- Problem: Both effects are declared `async`. Solid's dependency tracking captures only the signal reads that happen synchronously **before** the first `await`. The first effect reads `props.visible` synchronously (tracked) but reads `selectedDate()` *after* `await invoke(...)` — that read is NOT tracked. The second effect reads `selectedDate()`, `fromHour()`, `toHour()` synchronously (tracked), but the `setFilteredSlots` / `setTotalWords` happen post-await, which is fine. However, if a follower-edit ever moves a tracked read past the `await`, it silently breaks. Today the `if (d.length > 0 && !selectedDate())` line in F1's first effect is a footgun: `selectedDate()` happens after an `await`, so the effect won't re-run when `selectedDate` changes. (It only happens to work because the effect re-runs when `props.visible` flips.)
- Proposed fix: Read all dependencies synchronously into locals at the top of the effect; do the IO with a non-tracked async function (`createResource` or a plain `async` helper invoked from a synchronous effect body). Pattern: `createEffect(() => { const visible = props.visible; if (!visible) return; void loadDates(); });`
- Risk: Low; fixes a latent bug that would surface during the planned M19 refactor.

### F2. `setTimeout` for "copied" badge leaks if the component unmounts mid-flight [MEDIUM]
- Location: `src/components/HourSlotCard.tsx:23` (`setTimeout(() => setCopied(false), 1500)`) and `src/App.tsx:92` (`setTimeout(() => setCopiedAll(false), 2000)`).
- Problem: Neither timeout is registered with `onCleanup`. If the user clicks Copy and the card disappears (search query change, filter apply, route swap during HMR), the timer still fires `setCopied(false)` on a disposed signal, producing a "computation rendered after dispose" warning in dev and a wasted update in prod. The 1500/2000 ms inconsistency between the two badges is also a UX nit.
- Proposed fix: Track the timeout id and clear it in `onCleanup`, or replace with a small `useTimeout` helper. Standardize the duration (1500 ms).
- Risk: None.

### F3. Module-scope mutable scroll state survives HMR and leaks across navigation [MEDIUM]
- Location: `src/components/Timeline.tsx:7` (`let savedTimelineScroll = 0`) and `src/components/HourSlotCard.tsx:14` (`const scrollPositions = new Map<number, number>()`).
- Problem: Both globals work today because the app has a single timeline view. But (1) the `Map` grows unboundedly — every slot id ever rendered stays forever, creating a slow memory leak over a long session; (2) HMR may double-instantiate the module, doubling the state; (3) once `viewMode` (M18) lands, switching between search/filter/default views will surprise users with stale scroll restores. M22 plans a store — fold these into it with explicit eviction (LRU or "drop ids not in current `slots()`").
- Proposed fix: Move both into the planned `appStore`. Bound the Map (e.g., last 100 slot ids) and clear it on filter/search transitions.
- Risk: Low. Behavior change is intentional.

### F4. Redundant 5 s timeline poll on top of `transcription-updated` event [MEDIUM]
- Location: `src/App.tsx:135-138` (`setInterval(loadTimeline, 5000)`) layered on top of the listener at `src/App.tsx:126-133`.
- Problem: The event-driven path already calls `loadTimeline()` whenever the backend emits. The 5 s timer therefore fires a redundant SQL+IPC round-trip every 5 s while the app is open — useless work and a guaranteed race against the event handler. Audit M17 calls out the 5-effects pattern but does not specifically name the redundancy of this poll vs. the event subscription.
- Proposed fix: Delete the 5 s interval entirely. The backend event is the source of truth; if a one-shot resync is desired, do it on `visibilitychange` → `visible`.
- Risk: None functionally (the event already covers this), low product risk if the event ever silently fails to fire — mitigate with a 60 s safety net rather than 5 s.

### F5. Status polling runs at 1 Hz even when window is hidden [MEDIUM]
- Location: `src/App.tsx:109-114` (`setInterval(loadStatus, 1000)`) and the local-elapsed ticker at `src/App.tsx:116-124`.
- Problem: The status invoke + the local-elapsed `setInterval` both keep ticking when the user has Cmd+H'd or minimized the window. On a battery this burns CPU + IPC for a UI no one is looking at. Tauri windows do not auto-suspend rAF the way browsers do.
- Proposed fix: Pause both intervals when `document.hidden` is true; resume on `visibilitychange`. Optional: replace `setInterval(1000)` with `requestAnimationFrame`-throttled-to-1Hz, so it naturally suspends when minimized in some platforms.
- Risk: Low. Verify the status bar still shows the correct elapsed time on re-show (call `loadStatus` once on `visibilitychange`).

### F6. `secondsRemaining()` and `progress()` are recomputed on every signal read with no memoization [LOW]
- Location: `src/App.tsx:146-147`, consumed at `src/App.tsx:175-176, 181`.
- Problem: Both are plain functions, not `createMemo`. They're invoked separately in three JSX positions, each creating its own tracking subscription to `status()` and `localElapsed()`. With the 1 s tick this is cheap but unnecessary. More importantly, the math `Math.min(1, localElapsed() / status().segment_duration_secs)` produces FP noise — successive renders may flap between e.g. 0.2499999 and 0.25, churning the SVG `stroke-dashoffset` even when no visible change occurred.
- Proposed fix: Wrap both in `createMemo` and round `progress` to 3-4 decimal places before passing to the SVG attribute.
- Risk: None.

### F7. Empty catch blocks silently swallow IPC errors [MEDIUM]
- Location: `src/App.tsx:49, 58, 64, 72, 76`, `src/components/FilterPanel.tsx:27, 46`, `src/components/AudioLevelBars.tsx:38`.
- Problem: Every `invoke()` is wrapped in `try { ... } catch (_) {}`. A backend panic, a missing-command typo, or an `AppError` (per planned H7) becomes invisible — no console log, no toast, no user signal. This violates the "no empty catches" anti-pattern in the global rules and will dramatically slow debugging once the planned `AppError` migration lands.
- Proposed fix: Replace each empty catch with at minimum `console.error("[invoke X]", e)`. Better: a tiny `safeInvoke` wrapper in the planned `src/api.ts` (H6) that logs + optionally shows a toast on `AppError.kind === "Internal"`.
- Risk: None.

### F8. `searchQuery` input handler races with the 200 ms debounce on clear [MEDIUM]
- Location: `src/App.tsx:154` (`onInput={(v) => { setSearchQuery(v); if (!v) clearFilter(); }}`) interacting with `src/App.tsx:140-144` (the debounce effect).
- Problem: When the user clears the input, `setSearchQuery("")` fires the debounce effect (which schedules `searchSlots("")` in 200 ms) AND `clearFilter()` synchronously calls `loadTimeline()`. Then 200 ms later `searchSlots("")` runs again, doing a second `get_timeline` invoke. Two round-trips for one keystroke. Worse, if the user types another character within the 200 ms window after clearing, the in-flight `clearFilter()` → `loadTimeline()` resolves *after* the new `searchSlots(v)` and overwrites the search results with the timeline.
- Proposed fix: Drop the `if (!v) clearFilter()` shortcut — let the debounce handle the empty case via the existing branch in `searchSlots`. Or, if the immediate clear is desired, cancel the in-flight debounce by resetting the signal in a microtask.
- Risk: Low; preserves UX for the common case (typing) and removes the race.

### F9. `subscribe_audio_level` channel is never told to stop and AudioLevelBars mounts twice [MEDIUM]
- Location: `src/components/AudioLevelBars.tsx:28-45`. Component is mounted twice simultaneously: once inside `Timeline` empty state (`Timeline.tsx:61`) and once unconditionally in `StatusBar` (`StatusBar.tsx:37`).
- Problem: (1) Each mount opens a new `Channel` and `await invoke("subscribe_audio_level", { channel })`. The Rust side spawns a 33 ms polling task per subscription (audit M7). With two subscribers active the backend is doing 2× the work; if the user opens/closes the empty-state Timeline several times via filter/search transitions, channels accumulate because `onCleanup` only cancels the rAF — there is no `unsubscribe_audio_level` invoke and no signal to the backend that the channel has been dropped. (2) The `setDisplayLevel((prev) => ...)` self-reference inside the rAF callback is fine in Solid (no batching surprise) but reads stale `targetLevel` if the backend pushes faster than rAF — that's by design, but worth a comment.
- Proposed fix: Add an `unsubscribe_audio_level` command (or have `subscribe_audio_level` return a token that the frontend can call back). Call it in `onCleanup`. Alternatively, lift the audio-level subscription to a single store-level subscription used by both consumers (pairs naturally with M22). Confirm `Channel` close semantics in Tauri v2 — last I checked, dropping the JS-side `Channel` does not signal Rust.
- Risk: Medium. Needs a small Rust-side change. Without it, the count-of-subscribers grows over the session.

### F10. Filter overlay has no Escape-to-close, no focus trap, no aria roles [LOW]
- Location: `src/components/FilterPanel.tsx:67-118`.
- Problem: The overlay closes on backdrop click but ignores Escape. Tab navigation can leak to the underlying timeline (no focus trap). The overlay is not announced as a dialog (`role="dialog"`, `aria-modal="true"`, `aria-labelledby` missing). Search input (`SearchBar.tsx:16-22`) has no `aria-label`. Pause button (`StatusBar.tsx:41-43`) has only visible text but no semantic state. The whole app has no keyboard shortcuts (e.g., Cmd+F to focus search, Cmd+/ to toggle filter), which the audit doesn't cover.
- Proposed fix: Add `onKeyDown` Escape handler to the overlay, set `tabindex` and focus the panel on open, add aria attrs. Wire global Cmd+F via `window.addEventListener("keydown", ...)` registered/cleaned in `App.tsx`.
- Risk: None.

### F11. Most exported functions/components lack explicit return types [LOW]
- Location: across `src/App.tsx`, `src/components/*.tsx`. E.g., `App.tsx:31` (`export default function App()`), `App.tsx:43` (`async function loadTimeline()`), `App.tsx:146-147` (the two derived functions), `FilterPanel.tsx:13`, `AudioLevelBars.tsx:4`. Only the utility functions in `src/utils/format.ts` annotate returns.
- Problem: TS `strict: true` is on but the global TS rule says "Use explicit return types on exported functions". Inferred return types are fine for internal helpers, but exported component functions, async invokers, and derived signals are exactly where drift bites you (e.g., a future change to `searchSlots` accidentally returning a value would not be caught by callers expecting `void`).
- Proposed fix: Add `: JSX.Element` to component returns and `: Promise<void>` to the async helpers. Trivial.
- Risk: None.
# Pass-2 Review: Frontend Types & API Surface

## Summary

The IPC surface works but its contract lives in two unsynchronized places: `commands.rs` (Rust payload structs + raw `Result<T, String>`) and hand-written interfaces inside `App.tsx`. Even without adopting tauri-specta, several incremental fixes would shrink the drift surface meaningfully — extracting `types.ts`, mirroring the `events.rs` constant on the TS side, dropping the `HourSlotPayload` duplicate, deleting the now-dead Rust `copy_to_clipboard` command, and replacing `} catch (_) {}` with a tiny logging wrapper. The audited H5/H6/H7/M3 items are explicitly skipped here.

## Findings

### TY1. `interface HourSlot` and `interface AppStatus` live inside `App.tsx` [HIGH]
- Location: `src/App.tsx:10-29`; consumers `src/components/HourSlotCard.tsx:2`, `src/components/Timeline.tsx:4`, `src/components/StatusBar.tsx:3`, `src/components/FilterPanel.tsx:3`.
- Problem: Four leaf components reach back **up** into `../App` just to import a type. This creates a circular conceptual dependency (components depend on the page that hosts them) and makes future moves of `App.tsx` painful. It also blocks the audited H6/H5 work because `api.ts` will need these types — and importing them from `App.tsx` would create a real cycle.
- Proposed fix: Move both interfaces into `src/types.ts` (a new file) and re-export from `App.tsx` only if needed for back-compat. One mechanical pass — no logic change. This is also the natural target for the H5 generated bindings if/when adopted (drop-in replacement).
- Risk: None. Pure refactor verified by `tsc --noEmit`.

### TY2. `} catch (_) {}` swallows every IPC error silently [HIGH]
- Location: `src/App.tsx:49, 58, 64, 72, 76, 80`; `src/components/FilterPanel.tsx:27, 46`; `src/components/AudioLevelBars.tsx:38`.
- Problem: Every `invoke()` failure — including the `Result<T, String>` error string Rust deliberately produces — is discarded. A user reporting "the timeline is stuck" gives us nothing in the devtools console. This survives even without H6: no api.ts needed.
- Proposed fix: Add a thin `tryInvoke<T>(cmd, args)` helper in `src/utils/invoke.ts` that wraps `invoke<T>` in try/catch and logs the failure with the command name (`console.warn("[ipc]", cmd, err)`), returning `null` or rethrowing per call site. Replace the empty catches. ~20 LOC, no API surface change.
- Risk: None — strictly additive. The existing `as any`-style suppression goes away.

### TY3. `Channel<{ level: number }>` declared inline; canonical shape is `AudioLevelEvent` in Rust [MEDIUM]
- Location: `src/components/AudioLevelBars.tsx:31`; canonical `src-tauri/src/commands.rs:49-52` (`AudioLevelEvent { level: u32 }`).
- Problem: The TS `Channel<...>` payload type is inlined as an anonymous object literal. If Rust adds `peak: u32` or renames `level`, TS keeps compiling silently and the bar reads `undefined`. The Rust side already has the canonical type — TS is just not using it.
- Proposed fix: Declare `interface AudioLevelEvent { level: number }` in `src/types.ts` (alongside TY1) and import it: `new Channel<AudioLevelEvent>()`. Comment-link the Rust struct line for grep-ability. Doesn't require codegen.

### TY4. Event name `"transcription-updated"` hardcoded in TS while Rust has `events::TRANSCRIPTION_UPDATED` [MEDIUM]

### TY5. `HourSlotPayload` duplicates `storage::HourSlot` field-for-field [MEDIUM]
- Location: `src-tauri/src/commands.rs:12-35` vs `src-tauri/src/storage.rs:10-19`.
- Problem: `storage::HourSlot` already derives `serde::Serialize`. The `HourSlotPayload` struct + the `From<storage::HourSlot>` impl + the `.into_iter().map(Into::into).collect()` calls in 3 commands (`commands.rs:63, 72, 111`) exist purely to forward identical fields. The "boundary type" pattern is justified when shapes diverge; here, they don't.
- Proposed fix: Delete `HourSlotPayload`, change command return types to `Vec<crate::storage::HourSlot>`, drop the `From` impl and the `.map(Into::into)` calls. The `#[serde(rename_all)]` posture can be preserved by adding the same attribute on `storage::HourSlot` if needed (it isn't today — both use the default snake_case). The frontend `HourSlot` interface is unaffected.
- Risk: Low — deliberate behavior-preserving cleanup. The argument *for* keeping the boundary type is "future divergence" but YAGNI applies; reintroduce it the day Rust needs a field the frontend shouldn't see.

### TY6. `copy_to_clipboard` Rust command is dead in 99% of cases [MEDIUM]
- Location: `src-tauri/src/commands.rs:144-164`; caller `src/App.tsx:80` (`try { navigator.clipboard.writeText } catch { invoke("copy_to_clipboard") }`).
- Problem: The frontend tries `navigator.clipboard.writeText` first; the Tauri webview has clipboard access in default config, so the catch branch never fires under normal use. The Rust command exists as an untested fallback nobody exercises. Two code paths, async ordering subtly different (sync error vs. promise rejection), one effectively dead.
- Proposed fix: Pick one. Either (a) delete `copy_to_clipboard` from `commands.rs` and `lib.rs` registration, simplify `App.tsx:80` to a single `await navigator.clipboard.writeText(text)` with a logged catch — or (b) if pbcopy is the deliberate macOS-only fallback for a known failure mode, document it (`// fallback when webview lacks clipboard-write permission, e.g. CSP-locked builds`). Recommend (a) — the audit's M20 will eventually add a CSP, and the Tauri clipboard plugin (`@tauri-apps/plugin-clipboard-manager`) is the canonical answer if webview write breaks.
- Risk: Low. Verify on a clean build that the webview path actually works before deletion.

### TY7. Snake_case ↔ camelCase param mapping is unverifiable at compile time [MEDIUM]
- Location: `src/components/FilterPanel.tsx:38-42` passes `{ fromKey, toKey }`; Rust `commands.rs:107-108` declares `from_key: String, to_key: String`. Tauri auto-converts camelCase → snake_case, so this works — but no test pins it.
- Problem: Tauri's auto-conversion is undocumented per-call and changes between major versions. If a new command is added with a multi-word param like `audioPath`, a contributor can't tell from the call site whether it should be `audioPath` or `audio_path`. Today there's exactly one such param pair — easy to fix before there are ten.
- Proposed fix: Until H6 lands, add a one-line comment block at the top of `commands.rs` documenting the convention: "Frontend invokes use camelCase; Rust params use snake_case; Tauri auto-converts. See https://v2.tauri.app/develop/calling-rust/#passing-arguments." Pair with TY2's logging wrapper — an IPC failure would now print the offending command name, making mismatches diagnosable in seconds rather than minutes.
- Risk: None (doc-only).

### TY8. `MAX_TIMELINE_LIMIT` clamping is silent [LOW]
- Location: `src-tauri/src/commands.rs:60` (`limit.clamp(1, MAX_TIMELINE_LIMIT)`).
- Problem: A frontend that asks for `limit: 1000` gets back at most 200 rows with no signal. This is a frontend concern: pagination logic that infers "more pages exist" from `result.length === requested` will deadlock. Today only `App.tsx:47` requests `50` so the bug is latent — but it'll fire the day someone adds infinite scroll.
- Proposed fix: Two options. (a) Return an envelope `{ slots: Vec<HourSlot>, applied_limit: i64 }` so the frontend knows what was actually used. (b) Cheaper: if `limit > MAX_TIMELINE_LIMIT`, return `Err(format!("limit {limit} exceeds max {MAX_TIMELINE_LIMIT}"))` so the bug surfaces loudly during dev. Recommend (b) for now; (a) when pagination lands.
- Risk: (b) is none — current callers all pass 50. (a) is a payload shape change.

### TY9. `AppStatus.is_recording` is `!is_paused` server-side — UI re-derives the same booleans into a label [LOW]
- Location: `src-tauri/src/commands.rs:80-81` (`is_recording = !is_paused`); `src/components/StatusBar.tsx:13-25` (computes `stateLabel` and `dotClass` from the boolean pair plus `is_transcribing`).

The TS recomputes the state from the booleans in two parallel ladders (`stateLabel`, `dotClass`) — a classic enum-encoded-as-flags smell. `is_recording = !is_paused` is also a lie at startup before the audio engine is ready (it reports `is_recording: true`, hence the "Starting up..." branch in `Timeline.tsx:82` keying on `!isRecording && !isPaused` is currently unreachable).
- Proposed fix: Add a `pause_state: "none" | "manual" | "system"` field (already represented by `state::PauseReason` in Rust — just `serde` it) and let the UI compute labels from `pause_state` plus `is_transcribing`. Keep `is_recording`/`is_paused` for one release as deprecated to avoid breaking the layout. Pair this with a future `engine_state: "starting" | "ready" | "stopped"` to make "Starting up..." reachable.
- Risk: Low — additive field. UI changes are local to `StatusBar.tsx` and `Timeline.tsx`.

### TY10. `FilterPanel` calls `invoke()` directly with no shared error path [LOW]

# Pass-2 Review: CSS & Design System

## Summary
The stylesheet is a single 544-line dark-mode-only file that loads a full Tailwind v4 runtime which is never actually consumed (no utility classes anywhere in TSX). Beyond the audit's deferred tokenization/buttons/scrollbars/typography/animation items, the bigger gaps are accessibility (no `prefers-reduced-motion`, no `:focus-visible` outlines on interactive elements, low-contrast small text on a transparent vibrancy backdrop), unused dead rules left from the deleted `TranscriptionCard`, and a hand-painted RGBA tray icon that could be a static SVG/PNG asset.

## Findings

### CS1. Tailwind is imported but no utility classes are used anywhere [high]
- Location: `src/styles.css:1` (`@import "tailwindcss";`), `vite.config.ts:3,8` (`@tailwindcss/vite` plugin), `package.json:12,15` (`@tailwindcss/vite`, `tailwindcss` deps).
- Problem: A full Grep across `src/**` finds only the import in `styles.css` itself plus the substring "card-text-scroll" — zero `class="..."` strings use any Tailwind utility (no `flex`, `grid`, `p-`, `m-`, `text-`, `bg-`). The Tailwind preflight reset is shipping in the bundle anyway and (importantly) injects opinionated base styles (margin/padding resets, `*::before/::after` borders, etc.) that conflict with the hand-tuned skeuomorphic look. It's also adding ~10 KB of unused CSS and a build-time scan.
- Proposed fix: delete the `@import "tailwindcss";` line, drop `@tailwindcss/vite` from `vite.config.ts`, remove both deps from `package.json`. If you want utility-first later, re-introduce intentionally with a config that scopes preflight off.
- Risk: low — verify visually that removing preflight doesn't expose any styles that were silently relying on it (margin: 0 on `<button>`, etc. — most are already overridden in `styles.css`).

### CS2. No `prefers-reduced-motion` handling — spinners and animations run unconditionally [high a11y]
- Location: `styles.css:249` (`.transcribing-spinner` spin), `:263` (`.transcribing-spinner-large` spin), `:266` (`@keyframes spin`), `:490` (`.status-dot.recording` `pulse-dot 2s`), `:516` (`.status-dot.transcribing` spinning), `:210` (`.countdown-ring-progress` `transition: stroke-dashoffset 1s linear`), plus `transform: translateY(-0.5px)` card hover at `:288`.
- Problem: Users who set "Reduce motion" at the macOS level still see the recording dot pulse forever and spinners spin — a known vestibular accessibility issue. Apple HIG explicitly asks apps in vibrancy windows to honor this.
- Proposed fix: add a single block at end of file:
  ```css
  @media (prefers-reduced-motion: reduce) {
    *, *::before, *::after {
      animation-duration: 0.001ms !important;
      animation-iteration-count: 1 !important;
      transition-duration: 0.001ms !important;
    }
  }
  ```
  Optionally swap the recording dot's pulse for a static colored dot instead of crushing duration.
- Risk: none — only affects users who opted in.

### CS3. No `:focus-visible` outlines on interactive elements (keyboard a11y) [high a11y]
- Location: every button — `.card-copy-btn` (`:314`), `.pause-btn` (`:520`), `.filter-btn` (`:427`), `.filter-chip` (`:406`), `.filter-toggle-btn` (`:460`), `.filter-close` (`:397`), `.filter-clear-btn` (`:450`), `.drag-handle` (`:17`). Only `.search-input:focus` (`:89`) defines a focus ring.
- Problem: Tab-navigating the UI shows no visible focus indicator. Combined with `body { -webkit-user-select: none }` (`styles.css:11`) and the dark theme, a keyboard user is completely lost.
- Proposed fix: add a global focus-visible style early in the file:
  ```css
  :focus-visible {
    outline: 2px solid rgba(80,160,255,0.7);
    outline-offset: 2px;
    border-radius: inherit;
  }
  button:focus:not(:focus-visible) { outline: none; }
  ```
- Risk: low visual change. Coordinate with M13 button refactor.

### CS4. Color contrast on small ancillary text fails WCAG AA against vibrancy [medium a11y]
- Location: `.card-segments` `rgba(255,255,255,0.2)` 10 px (`:309-313`), `.card-device` `rgba(255,255,255,0.15)` 10 px (`:364-368`), `.status-device` `rgba(255,255,255,0.18)` 11 px (`:517`), `.empty-subtitle` `rgba(255,255,255,0.25)` 12 px (`:174-180`), `.countdown-label` `rgba(255,255,255,0.20)` 11 px (`:224-227`).
- Problem: Body is `background: transparent` (`:13`) — actual contrast depends on whatever the user has behind the window via macOS vibrancy. At alpha 0.15-0.20 over a mid-tone desktop, contrast ratio is well under 3:1, often unreadable. 10 px is also below Apple HIG minimum.
- Proposed fix: either bump alpha to ≥ 0.45 for these labels, or render them on top of a small opaque pill (already done for `.device-badge` — extend the pattern). Bump 10 px text to 11 px minimum.
- Risk: low; it's a readability win.

### CS5. Dead CSS — three rules with no matching markup [medium]
- Location: `styles.css:251-264` `.transcribing-large` and `.transcribing-spinner-large` (no usage in any `.tsx` — only the small spinner is used in `Timeline.tsx:37`); `styles.css:144-146` `.empty-state .audio-level-container` overrides padding for an audio-level-bars-inside-empty-state combo that exists at `Timeline.tsx:61` but has only one consumer (worth keeping but document it).
- Proposed fix: delete `.transcribing-large` + `.transcribing-spinner-large` entirely. Pairs with H3 "remove dead code" cleanup philosophy.
- Risk: none.

### CS6. `:root { color-scheme: dark }` hard-codes dark mode despite vibrancy being adaptive [medium]
- Location: `styles.css:3`, `tauri.conf.json` window vibrancy config.
- Problem: macOS vibrancy windows adapt to the system appearance — when the user is in light mode, vibrancy renders a near-white blurred backdrop. This stylesheet still paints white-on-white text (`rgba(255,255,255,0.82)`) over it, making the entire UI invisible. `color-scheme: dark` at `:root` only affects form-control rendering and the scrollbar; it does not protect against the user's chosen mode.
- Proposed fix: either (a) force the Tauri window to dark vibrancy material in Rust regardless of system theme (cleanest — matches today's hard-coded look), or (b) add a `@media (prefers-color-scheme: light)` block that flips the foreground tokens. Option (a) is one-line and matches current intent.
- Risk: medium if you go with (b) — needs a full token pass. (a) is the strategic choice given the rest of the stylesheet is dark-only.

### CS7. Z-index management is ad-hoc — single `z-index: 100` with no scale [low]
- Location: `styles.css:378` (`.filter-overlay { z-index: 100 }`). No other z-index in the file. The transcribing banner (`:230-242`) is in normal flow but renders above cards via DOM order, the filter active bar (`:444`) is similarly stacking-context-only, and the macOS tray context menu lives in a separate native window (no collision).
- Problem: Today there's exactly one stacking layer so `100` is harmless, but as soon as a second overlay is added (toast, modal, dropdown) you'll need a scale. Also `.filter-overlay` has `position: fixed; inset: 0` over the whole window including the 28 px drag handle — clicking the overlay near the top can't drag the window, which is surprising.
- Proposed fix: add CSS custom-property scale (`--z-overlay: 100; --z-toast: 200; --z-modal: 300`) when M12 lands. For the overlay-vs-drag-handle issue, give `.filter-overlay` `top: 28px` (or render it inside the timeline container).
- Risk: none.

### CS8. Tray icon is 90 lines of pixel-painting Rust — should be a static asset [medium]
- Location: `src-tauri/src/tray.rs:65-146` (`make_icon` — pixel-by-pixel rasterization of a page-with-waveform glyph at 44×44).
- Problem: Re-rasterizes on every app launch, hard to iterate (no preview, can't tweak in a vector editor), and the 5-bar waveform at 44 px is jaggy. The repo already has `src-tauri/icons/` with the app icon assets — adding `tray-icon@2x.png` (or a `tray.svg` shipped as bytes) lets a designer replace the glyph without touching Rust.
- Proposed fix: design once in SVG, export `tray.png` (22×22) and `tray@2x.png` (44×44), `include_bytes!("../icons/tray@2x.png")` in `tray.rs`, pass to `Image::from_bytes`. Delete `make_icon`.
- Risk: low — pure refactor with visual parity if exported well.

### CS9. `AudioLevelBars` hard-codes pixel sizes — chunky on @1x, wrong size in different contexts [low]
- Location: `src/components/AudioLevelBars.tsx:5-9` (`barCount=7, barWidth=4, gap=3, maxHeight=32`); used in two places: `StatusBar.tsx:37` (small bar) and `Timeline.tsx:61` (large empty-state bar). Both consumers get identical sizing.
- Problem: Audit L12 already noted parameterization. Adding to it: `barWidth: 4, gap: 3` in absolute pixels means at @1x (external monitor) the bars are visibly chunky vs at @2x retina they look right. SVG with `viewBox` already scales — driving width/height from CSS via parent sizing would let the empty-state version scale up naturally.
- Proposed fix: accept `width`/`height` props (or a `size: "sm" | "lg"` enum), use viewBox scaling. Pairs with L12.
- Risk: none.

### CS10. `.drag-handle` is the only draggable region — no escape hatch on interactive children [low]
- Location: `styles.css:17-26` (`-webkit-app-region: drag` on the 28 px strip). No `-webkit-app-region: no-drag` declared anywhere. `.drag-dots span` are 4×4 px (`styles.css:32-37`) — a 5-dot row with 4 px gaps gives ~36 px tap target, well below Apple HIG's 44 pt minimum. Also `cursor: grab` (`:26`) on the whole strip is fine, but the dots are decorative-only and `pointer-events` should probably be `none` on them so clicks reach the drag region cleanly.
- Problem: As the UI grows (e.g. add a settings cog inside the drag handle later), child interactive elements will be uncliccable until each one gets `app-region: no-drag`. Worth establishing the pattern now.
- Proposed fix: add `.drag-dots { pointer-events: none }` and document a `.no-drag { -webkit-app-region: no-drag; app-region: no-drag }` utility for future buttons placed inside the title bar.
- Risk: none.

### CS11. Mixed unit usage across the file — pixels everywhere except line-height and viewport-height [low]
- Location: every `padding`, `font-size`, `border-radius`, `gap` is in `px` (e.g. `:73-78`, `:101`, `:218`, `:273-275`); the only non-px values are `line-height: 1.6` (unitless, correct), `height: 100vh` (`:46`), and `max-width: 260px`. No `rem`/`em` anywhere.
- Problem: The app does not respect the user's browser/system base font-size preference (which Safari and the WKWebView honor for accessibility). All-pixel sizing was a deliberate Apple-HIG-style choice for a fixed-size utility window, so this is borderline acceptable — but worth flagging because you'll want `rem` for the typography scale when M12/L20 lands so a future "size scale" preference is one variable away.
- Proposed fix: when tokenizing in M12, define `--text-xs` etc. in `rem` so the scale composes with user font-size. No need to change anything before that.
- Risk: none until the tokenization happens.
