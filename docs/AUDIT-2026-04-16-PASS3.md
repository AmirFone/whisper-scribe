## How this audit was produced

After pass-2 landed (25+ audit items applied, 174 tests passing, both Rust and frontend builds clean), 10 specialized review agents were dispatched in two overlapping waves (Sonnet model to avoid Opus rate limits). Each had a narrow scope, was given the four prior docs as context, and was instructed not to re-flag known items. **All 10 agents completed.** Two scopes (Build/Security + Testing Strategy) were also re-run on the pre- and post-refactor code; this document uses the post-refactor versions.

## Top-of-mind issues across reviewers

Ten patterns showed up in multiple agent reports independently. Tackle them in this order:

1. **Orphan dedup is semantically broken** (A5 + S14 + T7). `storage.start_time` is set to the transcription *completion* time at append time, but `pipeline::already_transcribed` compares against the *capture* time parsed from the WAV filename. These two clocks have no relationship. Every orphan fails the check and is re-transcribed on every restart. The doc comment in `has_transcription_near` acknowledges "only catches orphans that were the FIRST segment of their hour" — that is a symptom of the deeper mismatch. One-line fix: `storage::append_to_hour_slot` takes the WAV capture time from `TranscriptionResult::start_time` (which already parses the filename) and stores *that* as `start_time`. The roundtrip then holds.

2. **The `segment_started_at` mutex is a deadlock trap** (C1 + AU-NEW-1 + AU-NEW-2 + AU-NEW-3). Pass-2 moved `SEGMENT_STARTED_AT` from a module static to an `Arc<Mutex<Option<DateTime<Utc>>>>` on `AppState`. The audio callback now holds `state.lock()` outer AND `segment_started_at.lock()` inner on every paused buffer (~93× per second at 48 kHz / 512 frames). The ordering is undocumented, the locks are unnecessary for a 64-bit value, and the pause path stamps `Utc::now()` hundreds of times per second. Fix: replace with `Arc<AtomicI64>` (epoch millis; `i64::MIN` = unset). This is the "deferred C2" from pass-2 audit — it is now mandatory, not optional.

3. **`subscribe_audio_level` is an architectural anti-pattern** (C2 + C9 + A3 + F1). Per-mount Tokio tasks with no cancellation, no dedup, no limit, doubled by the Timeline+StatusBar co-mount. The fix (push events via `Emitter::emit`) was deferred in pass-1 (M7) and pass-2 (A9). Ship it next — every related finding dissolves.

4. **Graceful shutdown remains unreachable** (C5 + C6 + T2 + A9). Stderr forwarder thread, power monitor thread, and `is_transcribing` flag all leak past Tauri teardown; `Transcriber::drop` kills but never `wait()`s → guaranteed zombies. `C4 + AU7` was deferred in pass-2 as "architectural"; until it lands every production build ships a resource leak.

5. **Typed error contract is still missing** (A10 + F6 + T9 + TY-G). Every module returns `Result<T, String>`. Callers cannot branch on error kind. The frontend's `tryInvoke` collapses "IPC unreachable" and "empty success" into `null`. A single `src-tauri/src/error.rs` with a `WscribeError` enum unblocks several deferred items (graceful-shutdown routing, S2 idempotency, T1 protocol versioning).

6. **Dead capabilities / dependencies clutter the surface** (BS1 + BS2 + BS5 + BS10). `staticlib` artifact, `tauri-plugin-shell` live with `shell:allow-open` and zero callers, `rtrb` compiled but unused, `tokio::full` pulling every subsystem. One focused Cargo/config pass removes ~500 KB + one latent security surface.

7. **Zero CI, seven integration-test files testing a dead schema** (TS1 + TS3 + TS4 + TS10). `tests/storage_integration.rs` et al. all create their own `CREATE TABLE transcriptions` (the legacy schema) and never import the real `Storage` API. 174 tests pass but `Storage::append_to_hour_slot` has no regression guard. No CI exists.

8. **`AppState` is drifting toward god-object** (A1 + A7 + A2). `pub storage`, `pub transcriber: Mutex<Option<...>>`, `pub is_transcribing: AtomicBool` — all raw public fields reachable by any module. Delegation methods collapse the surface and enable the deferred A7 (transcriber-out-of-AppState) as a one-file change.

9. **Filter-view + transcription event collision** (F3). `loadTimeline()` early-returns when `filterActive()` is true, so every `TRANSCRIPTION_UPDATED` event silently discards new segments from the visible filter view. Correctness bug for anyone who records with a filter open.

10. **Hot-path callback work on paused buffers** (AU-NEW-2 + AU-NEW-3 + CS-N2). "Fast-path pause gate" takes the state mutex + stamps `Utc::now()` on every paused buffer; `.status-dot.transcribing` spins a rotationally-symmetric circle for zero visible effect. Pause-onset edge detection fixes the backend; dropping the spin animation fixes the UI.

Beyond those, the highest-impact NEW items are:
- `open_new_segment` failure leaves `writer=None` with no user signal — audio silently dropped (AU-NEW-4).
- Two pre-existing TS errors are trivially fixable (TY-C `String(circumference)`, TY-D `vite-env.d.ts`).
- `read_line_bounded` has a latent consume-bytes-permanently bug that is invisible to current tests (T1).
- `WSCRIBE_PYTHON` env override introduces an unauthenticated arbitrary-executable path (BS4).
- `FilterPanel` range-picker fires an IPC call on every signal change with no debounce (F2).
- `env_logger::init()` with no default filter silently discards all `log::info!`/`warn!` in packaged builds — amplified now that Python stderr is routed through `log::warn!` (BS6).

# Pass-3 Review: Architecture & Module Cohesion

## Summary

Pass-2 landed cleanly: the global statics are gone, `audio_dir` owns the filename grammar, pause is wired end-to-end, and `toggle_pause` is a single-call method. The dependency graph is now acyclic. What remains are subtler structural issues introduced or exposed by pass-2's own work:

1. `AppState` is growing into a god-object — it now owns five distinct concerns through public fields, two Arc-returning "arc accessor" methods per concern, and two pub fields that let any caller bypass the module interface entirely (`storage`, `transcriber`). The module boundary problem has migrated from `audio_engine`'s globals into `AppState`.
2. `get_current_device_name()` re-enumerates all audio devices on every call and is called three times per transcription segment — once in `commands.rs::get_status` (every poll), twice in `transcriber::transcribe` — with no caching and no coherence with the device actually selected at engine start.
3. `subscribe_audio_level` is a long-running `loop` inside an `async` Tauri command — it blocks a Tokio worker for the app's entire lifetime.
4. `transcriber.rs::count_samples` opens and reads the WAV header for every segment, but `audio_engine` already knows the exact sample count (`samples_written`). This field is dead (`dead_code` warning) because nothing passes it out.
5. The orphan-recovery dedup check (`has_transcription_near`) only matches against `hour_slots.start_time` (the first segment's timestamp per hour), so any non-first orphan segment is silently re-transcribed and double-appended on restart.
6. `AudioEngine::new` takes five parameters and produces a single-field struct (`_stream`).
7. `events.rs` is a one-constant file.

## Findings

### A1. `AppState` exposes `storage` and `transcriber` as `pub` fields — bypasses all encapsulation [severity: High]

- Location: `src-tauri/src/state.rs:24-25`
- Problem: `pub storage: storage::Storage` and `pub transcriber: Mutex<Option<transcriber::Transcriber>>` are raw public fields. Any module can reach into `state.storage.append_to_hour_slot(...)` or `state.transcriber.lock().as_ref()...` directly — which is exactly what `pipeline.rs:79,88` and `commands.rs:47,55,65,93,100` do. There is no enforcement of the contract. Ousterhout calls this an information-hiding violation: the module's implementation (a `Mutex<Option<...>>` wrapping a process-spawning struct) is fully visible to every consumer. When the transcriber eventually moves off `AppState` (deferred A7), callers will need mechanical surgery because the access pattern is scattered.
- Proposed fix: Add thin delegation methods to `AppState` — `state.transcribe(path)` (called only by `pipeline`) and typed methods like `state.append_transcript(...)`, `state.get_timeline(...)`. At minimum, make both fields `pub(crate)` to limit exposure to the crate boundary.
- Risk: Medium. Touches every `state.storage.*` and `state.transcriber.*` call site, but it is mechanical.

### A2. `get_current_device_name()` re-enumerates all audio devices on every call — called 3× per segment [severity: Medium]

- Location: `src-tauri/src/device_manager.rs:49-54`, called at `commands.rs:64`, `transcriber.rs:105`, `transcriber.rs:115`
- Problem: Every call invokes `cpal::default_host().input_devices()`, iterates all input devices, filters by name keywords, and returns the best-match name. Called (a) in `get_status` on every 1-second UI poll, (b) twice per transcribed segment inside `Transcriber::transcribe`. Device enumeration can park for 5–20 ms on coreaudio. The device selected at engine start and the device reported by `get_current_device_name()` are independently computed — they can drift if an audio device disconnects between engine start and the next poll.
- Proposed fix: Cache the selected device name on `AudioEngine` construction (already in scope as `device_name` at line 70) and store it in `AppState` (a `parking_lot::RwLock<String>`). `get_status` and `transcriber` read the cached name. The cache is invalidated only if device hot-swap (AU10) lands.
- Risk: Low.

### A3. `subscribe_audio_level` occupies a Tokio worker thread for the app's lifetime [severity: Medium]

- Location: `src-tauri/src/commands.rs:104-116`
- Problem: The command is declared `async` and runs an infinite `loop { channel.send(...); sleep(33ms).await; }`. Tauri's command executor dispatches this to a Tokio worker pool thread. There is no way to cancel it from Rust (deferred C9), and if the frontend reconnects (e.g. hot-reload in dev), a new poll loop starts alongside the orphaned one.
- Proposed fix: Replace with a `tauri::AppHandle`-driven push: in a dedicated audio-level watcher spawned once at startup, emit `audio-level` events via `app.emit(...)` on every N-th audio callback tick using an `AtomicU64` counter. The frontend subscribes with `listen`. This eliminates the permanent worker.
- Risk: Medium. Frontend needs to switch from `Channel` to `listen`. Coordinates with frontend F9.

### A4. `count_samples` re-reads the WAV header to compute `end_time` — dead field on `RecordingState` [severity: Low]

- Location: `src-tauri/src/transcriber.rs:42-44`, `src-tauri/src/audio_engine.rs:48` (`samples_written`), `src-tauri/src/transcriber.rs:212-214`
- Problem: `Transcriber::transcribe` calls `count_samples(path)` which opens the WAV file purely to call `.duration()`. Used to compute `end_time` which is already `dead_code` (never read by any consumer). The actual sample count is already tracked in `RecordingState::samples_written` but never leaves the audio engine. This is a textbook "shallow interface forces the consumer to re-derive internal state" violation.
- Proposed fix: Delete `count_samples`, set `end_time = start_time` (unused), and remove the dead field. The `dead_code` warning disappears and one disk open is saved per segment.
- Risk: Low.

### A5. Orphan-recovery dedup silently re-transcribes non-first segments in any existing hour [severity: Medium]

- Location: `src-tauri/src/pipeline.rs:129-133`, `src-tauri/src/storage.rs:228-237`
- Problem: `already_transcribed` calls `storage.has_transcription_near(ts)` which executes `SELECT COUNT(*) FROM hour_slots WHERE start_time = ?1`. `start_time` is set exactly once per hour — when the *first* segment of that hour is inserted. All subsequent segments do not update `start_time`. So if a crash happens mid-hour, the orphan scan will find segments 2, 3, 4; `has_transcription_near` returns `false` for all; and all are re-transcribed and appended, doubling the hour's text. See also S14 and T7.
- Proposed fix (pairs with S14): change `start_time` to store the *capture* time (parsed from the WAV filename) instead of the completion time. Then the dedup roundtrip holds. Strategic fix is S2's segments table.
- Risk: Medium. Changes meaning of `start_time` for existing rows. Document in migration doc.

### A6. `AudioEngine::new` is a 170-line constructor; `AudioEngine` struct is a single-field wrapper [severity: Medium]

- Location: `src-tauri/src/audio_engine.rs:57-228`
- Problem: Takes 5 parameters, builds a cpal stream, constructs a resampler, pre-allocates two buffers, closes over a debug counter, and returns `Self { _stream: stream }` — a struct that only holds one field. The constructor is doing what a builder would do, without the staged-construction visibility.
- Proposed fix: Introduce an `AudioEngineConfig` struct (5 fields, no `Arc` complexity — just the typed parameters) that `AudioEngine::new` takes as one argument. Callers construct `AudioEngineConfig::from_state(&state)`.
- Risk: Low.

### A7. `is_transcribing` is a `pub AtomicBool` field accessed directly by two unrelated modules [severity: Low]

- Location: `src-tauri/src/state.rs:32`, read at `commands.rs:73`, written at `pipeline.rs:76,97`
- Problem: Raw `pub` field. `pipeline.rs` writes directly; `commands.rs` reads directly. No accessor method, no documentation of who owns the invariant. The `Ordering::Relaxed` on the write and read is technically sound for a best-effort UI indicator, but the intent is nowhere documented.
- Proposed fix: Add `state.set_transcribing(bool)` (with `Ordering::Release`) and `state.is_transcribing() -> bool` (with `Ordering::Acquire`). Mark the field `pub(crate)`. Documents the ordering intent.
- Risk: Low.

### A8. `tray.rs::make_icon` is 80 lines of imperative pixel-painting inlined into the tray module [severity: Low]

- Location: `src-tauri/src/tray.rs:61-142`
- Problem: Two distinct concerns in one file. The icon generation is ~80 lines of geometric rendering — well over half the file. It runs anew every launch with no `OnceLock` cache.
- Proposed fix: Extract `make_icon()` to `src-tauri/src/tray_icon.rs` or embed a pre-rendered 44×44 RGBA asset at compile time via `include_bytes!`. If procedural generation must stay, wrap in `OnceLock`.
- Risk: Low.

### A9. `is_transcribing` is set before acquiring the transcriber lock — panic between set/clear leaks the flag forever [severity: Medium]

- Location: `src-tauri/src/pipeline.rs:76-98`
- Problem: `is_transcribing.store(true)` at line 76 happens before `state.transcriber.lock()` is attempted. If a panic occurs between lines 76 and 97, `is_transcribing` stays `true` forever and the UI reports "transcribing" indefinitely.
- Proposed fix: Move `is_transcribing.store(true)` to after the lock succeeds. Better: wrap the flag updates in a RAII guard — `struct TranscribeGuard<'a>(&'a AtomicBool)` with a `Drop` impl that clears the flag.
- Risk: Low.

### A10. Error type is `String` across all public module interfaces — no typed error enum exists [severity: Medium]

- Location: `src-tauri/src/storage.rs:59,142,167,...`, `src-tauri/src/transcriber.rs:33,40`, `src-tauri/src/audio_engine.rs:64`, `src-tauri/src/pipeline.rs`
- Problem: Every public `Result` uses `Result<T, String>`. Callers cannot match on error kind. `pipeline` can't distinguish recoverable storage errors from fatal engine errors. The deferred C4 (graceful shutdown) and S2 (idempotency) both need error-kind routing.
- Proposed fix: Introduce `src-tauri/src/error.rs` with a `WscribeError` enum: `Storage(rusqlite::Error)`, `Audio(String)`, `Transcriber(String)`, `Io(std::io::Error)`. Implement `From` conversions. Use internally; convert to `String` at the Tauri command boundary.
- Risk: Medium. Mechanical but touches every `map_err(|e| format!(...))` call site.

# Pass-3 Review: Concurrency, Threading & Lifecycle

## Summary

The pass-2 refactor moved shared state off module globals and wired the audio callback's pause gate through an `Arc<AtomicBool>`. That eliminated the largest contention risks. What remains is a cluster of subtler but still consequential issues: a two-level lock acquisition sequence inside the paused callback branch creates a latent deadlock window; the `subscribe_audio_level` command spawns an unbounded number of infinite tokio tasks; the `is_transcribing` flag uses `Relaxed` ordering; three background threads (power monitor, cleanup timer, stderr forwarder) have no shutdown path; `process_orphans` races with the freshly-constructed audio engine on the same directory; and the daemon's stderr forwarding thread holds a `Child` handle that can block the OS from reaping the child process.

## Findings

### C1. Two-level lock acquisition in the paused callback branch [severity: high]

- Location: `src-tauri/src/audio_engine.rs:163-168`
- Problem: `let mut s = state_clone.lock(); s.resample_buf.clear(); *s.segment_started_at.lock() = Some(…);` — outer + inner lock held simultaneously. Same nesting at `:290` inside `open_new_segment`. `parking_lot::Mutex` is not reentrant. Any caller acquiring `segment_started_at` first and then `state` would deadlock. Today there is no such code path, but `elapsed_seconds` (called under `state.lock()`) locks `segment_started_at` — same order, so no deadlock now. The outer lock is being held unnecessarily across the inner lock, expanding contention window.
- Proposed fix: Eliminate the inner mutex entirely. Replace `Arc<Mutex<Option<DateTime<Utc>>>>` with `Arc<AtomicI64>` where `i64::MIN` means "no segment open". Write with `Ordering::Release`, read with `Ordering::Acquire`. This was deferred as C2 in pass-2 but is now pressing.
- Risk: Medium.

### C2. `subscribe_audio_level` spawns unbounded infinite tasks — no cancellation, no dedup [severity: high]

- Location: `src-tauri/src/commands.rs:104-116`
- Problem: Every frontend call spawns a Tauri-managed tokio task that loops forever calling `sleep(33ms)` and pushing to a `Channel`. There is no limit. Each call results in a separate infinite task. On frontend hot-reload or when `AudioLevelBars` mounts/unmounts/remounts, multiple polling loops pile up.
- Proposed fix: Stopgap: `Mutex<Option<tokio::task::AbortHandle>>` on `AppState`, abort any previous task before starting the new one. Better: the deferred A9/M7 push-event migration.
- Risk: Medium without A9.

### C3. `is_transcribing` written and read with `Relaxed` ordering — UI can observe stale false [severity: medium]

- Location: `pipeline.rs:76,97` (store), `commands.rs:73` (load)
- Problem: Both stores and the load use `Ordering::Relaxed`. On aarch64 this is a UI glitch; on weaker architectures it is a correctness exposure. Not covered by pass-2's C7 deferral (which was `pause_flag` only).
- Proposed fix: Release on store, Acquire on load. Zero overhead on aarch64. Pair with A7's accessor methods.
- Risk: None.

### C4. `process_orphans` and `AudioEngine::new` race on the audio directory [severity: medium]

- Location: `pipeline.rs:31` (`process_orphans`), `pipeline.rs:35-50` (engine construction)
- Problem: Sequential call order is correct today, but unguarded. A refactor could silently invert it; a fast-restart within the same second creates filename collisions that confuse the "skip latest" heuristic.
- Proposed fix: Add a comment asserting required ordering. Record orphan files at scan time (already done via returned `Vec`), consider adding `currently_writing: Option<&Path>` to `find_orphan_segments` (pairs with C10).
- Risk: Low.

### C5. Stderr forwarder thread holds `BufReader<ChildStderr>` — child may not be reaped on drop [severity: medium]

- Location: `transcriber.rs:164-169`
- Problem: Detached thread reads stderr line-by-line. When `Transcriber::drop` calls `child.kill().ok()`, the stderr thread continues blocking on `reader.lines()` until the child actually exits. `JoinHandle` is discarded. Delays OS-level resource reclamation.
- Proposed fix: Store the `JoinHandle<()>` in `DaemonHandle`. In `Drop`, kill the child, then `drop(daemon.reader)` to close the read end of the pipe and unblock the thread. Or use an `Arc<AtomicBool>` stop flag.
- Risk: Low.

### C6. `PowerMonitor` thread loops forever with no exit condition — `_handle` is never joined [severity: medium]

- Location: `power_monitor.rs:9-13`, `pipeline.rs:53`
- Problem: `PowerMonitor::new` spawns `loop { sleep(15s); ioreg … }`. The `_handle` is never joined; default `Drop` just detaches. The detached thread then holds `Arc<AppState>` indefinitely past Tauri teardown.
- Proposed fix: Add a shutdown channel / `Arc<AtomicBool>` stop flag. On `Drop`, set flag and join. Pairs with deferred C4/A6.
- Risk: Medium for broader shutdown design; Low for stop-flag alone.

### C7. `cleanup_old_segments` inside `process_orphans` can delete files the transcriber is about to open [severity: medium]

- Location: `pipeline.rs:126`
- Problem: After iterating orphans, `cleanup_old_segments(dir, MAX_RETAINED_SEGMENTS)` is called. If there are >6 orphans, some oldest are deleted — potentially files skipped but still legitimate.
- Proposed fix: Pass the set of successfully-processed (or skip-confirmed) paths to the cleanup rather than re-scanning.
- Risk: Low (requires >6 orphan files).

### C8. `AudioEngine` holds `Arc<AtomicU32>` (audio level) but `AppState` also holds the same `Arc` — dual-ownership with no designated writer protocol [severity: low]

- Location: `audio_engine.rs:134`, `state.rs:36`
- Problem: Both hold clones of the `Arc<AtomicU32>`. Nothing structurally prevents a future call site from writing from a second thread.
- Proposed fix: Add `#[doc]` comment on `AppState::audio_level_arc()` explicitly stating the cpal callback is the sole writer. Expose only `audio_level() -> u32` for readers; keep `audio_level_arc()` crate-private.
- Risk: None.

### C9. `subscribe_audio_level` tokio task holds `Arc<AtomicU32>` clone past Tauri runtime teardown [severity: low]

- Location: `commands.rs:108-115`
- Problem: During the 33ms sleep the `Arc` is still alive. Benign today; non-trivial if the `Arc` extends to shutdown-sensitive state.
- Proposed fix: `tokio::select!` with a cancellation token. Pairs with C2.
- Risk: None.

### C10. `find_orphan_segments` heuristic ("skip the newest file") is unsound when the engine has already created its first segment [severity: low]

- Location: `audio_dir.rs:97-103`
- Problem: `find_orphan_segments` returns everything except `segments[last]`. Correct today because `process_orphans` runs before engine construction. If called again with an engine running, returns the engine's in-progress segment as an orphan candidate.
- Proposed fix: Accept optional `currently_writing: Option<&Path>` parameter; explicitly exclude that path.
- Risk: Low.

# Pass-3 Review: Storage / SQL

## Summary

Post-pass-2, `storage.rs` is structurally sound: `prepare_cached` on all read queries, a proper PRAGMA bundle, `collect::<rusqlite::Result<Vec<_>>>()` everywhere. The deferred items (S1, S2/S3, S8, S9/S10) remain open. This pass surfaces new issues: the single `Mutex<Connection>` serializes reads WAL could parallelize; `prepare_cached` pool has an unchecked size bound; `sanitize_fts_query` passes single-char tokens; `has_transcription_near` uses un-cached `query_row` and produces false-negatives due to clock-meaning mismatch; `count()` and inline `query_row` calls are not cached; the FTS trigger fires on every update; `SEARCH_HOUR_SLOTS_SQL` hard-caps at 50; `foreign_keys=ON` is connection-scoped; `append_to_hour_slot` produces double-spaces on whitespace-leading text.

## Findings

### S11. Single `Mutex<Connection>` serializes all reads despite WAL enabling concurrent readers [medium]

- **Location:** `storage.rs:6-8`
- **Problem:** SQLite WAL mode allows multiple concurrent readers from separate connections. With a single `Mutex<Connection>`, all queries queue behind `append_to_hour_slot`. `commands.rs` exposes five read commands Tauri can invoke concurrently; a search (slow FTS) blocks all other reads + writes.
- **Fix:** `r2d2` + `r2d2_sqlite` connection pool (1 writer + N readers). Even a simple `Arc<(Mutex<Connection>, Mutex<Connection>)>` — one write, one read — helps.
- **Risk:** Medium.

### S12. `count()` and `has_transcription_near` bypass `prepare_cached` [low]

- **Location:** `storage.rs:228-243`
- **Problem:** Both use `query_row()` which calls `prepare` internally on every invocation. `count()` is called on every 1 Hz status poll.
- **Fix:** Two new `&'static str` consts + `prepare_cached`.
- **Risk:** None.

### S13. `sanitize_fts_query` produces valid-looking but FTS5-erroring output for single-char and dash-only tokens [medium]

- **Location:** `storage.rs:246-264`
- **Problem:** The filter allows `'-'` through. A word like `"-"` produces `"\"-\""`, which FTS5 on some SQLite builds rejects. Unicode-alphanumeric accepts CJK which FTS5's default tokenizer splits differently — zero-result queries with no diagnostic.
- **Fix:** `replace('"', "").replace('*', "")` + `filter(|w| w.chars().any(char::is_alphanumeric))`. Add tests for `"-"`, `"---"`, `"hello-world"`, CJK.
- **Risk:** Low.

### S14. `has_transcription_near` exact-string match has sub-second false-negative window [medium]

- **Location:** `storage.rs:228-237`, `pipeline.rs:133`
- **Problem:** `start_time` stores the transcription *completion* time (`timestamp.to_rfc3339()` where `timestamp` is the `DateTime<Utc>` from the result). The WAV stem is the *capture* time. These two timestamps come from different clocks and will never match. Every orphan fails dedup and re-transcribes on every startup. See A5 and T7 — same root cause viewed from three angles.
- **Fix:** Change `append_to_hour_slot` to take the capture timestamp from `TranscriptionResult::start_time` (already `audio_dir::parse_segment_timestamp`) and store that as `start_time`. One-line in `pipeline.rs`.
- **Risk:** Medium. Changes meaning of `start_time` for existing rows.

### S15. `append_to_hour_slot` produces double-spaces when input text has leading/trailing whitespace [low]

- **Location:** `storage.rs:155-161`
- **Problem:** `text = text || ' ' || excluded.text` unconditionally prepends a space. Whisper output often starts with `" Hello"`, producing double-spaces over time.
- **Fix:** `text = trim(text) || ' ' || trim(excluded.text)` in the SQL. Rust-side guard rejecting empty text.
- **Risk:** Low.

### S16. `SEARCH_HOUR_SLOTS_SQL` hard-caps results at 50 with no caller parameter [low]

- **Location:** `storage.rs:29-35`, `commands.rs:51-56`
- **Problem:** `get_timeline` supports caller-controlled `limit`/`offset`; `search_hour_slots` ignores both and hard-codes `LIMIT 50`. No pagination affordance.
- **Fix:** Parameterize the limit (`?2`) with `DEFAULT_SEARCH_LIMIT: i64 = 50, MAX_SEARCH_LIMIT: i64 = 200`. `prepare_cached` still works because the SQL string stays constant.
- **Risk:** Low.

# Pass-3 Review: Audio Engine

## Summary

Ten findings across `audio_engine.rs` and `audio_dir.rs`. The most severe is a lock-ordering violation in the pause path that can deadlock under parking_lot (AU-NEW-1). The second-most severe is unbounded `VecDeque` growth in `resample_buf` under certain sample-rate ratios (AU-NEW-5). Remaining findings: degenerate pause-path behaviour (CPU, correctness), silent data loss on `open_new_segment` failure, missing user signal on write errors, cleanup walking the directory twice per timer tick.

## Findings

### AU-NEW-1. Nested lock: `state` outer → `segment_started_at` inner — deadlock if reverse order ever occurs [high]

- **Location**: `audio_engine.rs:165-168`, `:290`
- **Problem**: `parking_lot::Mutex` is not reentrant. Lock order is `RecordingState` → `segment_started_at`. Future reverse ordering would deadlock silently on macOS release builds. See C1.
- **Fix**: Replace `Arc<Mutex<Option<DateTime<Utc>>>>` with `Arc<AtomicI64>` (`i64::MIN` = unset).
- **Risk**: Medium.

### AU-NEW-2. Pause callback acquires `state` mutex on every paused buffer — O(callbacks/sec) mutex traffic on the RT thread [medium]

- **Location**: `audio_engine.rs:163-169`
- **Problem**: Comment says "fast-path pause gate — pure atomic read" but the gate immediately takes `state_clone.lock()` on every paused callback (~93×/sec at 48 kHz/512 frames). Each call locks, clears `resample_buf` (already empty after first clear), overwrites `segment_started_at` with `Utc::now()` (syscall).
- **Fix**: Add callback-local `bool was_paused` tracking previous-buffer state. One-time work on pause onset only.
- **Risk**: Low.

### AU-NEW-3. `segment_started_at` written on every paused buffer [medium]

- **Location**: `audio_engine.rs:167`
- **Problem**: Consequence of AU-NEW-2. The UI's `get_status` shows "0 s elapsed" while paused because the timestamp is always ~now. Implicit and undocumented.
- **Fix**: Stamp once at pause onset (same fix as AU-NEW-2).
- **Risk**: None beyond AU-NEW-2.

### AU-NEW-4. `open_new_segment` failure leaves `writer=None` with no user-visible signal — audio silently dropped [high]

- **Location**: `audio_engine.rs:284-294`
- **Problem**: When `WavWriter::create` fails, `state.writer` remains `None`. `write_samples` silently returns early. `rotate_segment` sends nothing. The only signal is a `log::error!` that may never reach the user (see BS6). Can persist for the entire session.
- **Fix**: Emit a Tauri event (`"audio-error"`) using an `AppHandle` threaded via `RecordingState`, or set an `Arc<AtomicBool>` disk_error flag that `get_status` surfaces.
- **Risk**: Medium.

### AU-NEW-5. `resample_buf` VecDeque can grow past pre-allocated capacity [medium]

- **Location**: `audio_engine.rs:196-211`
- **Problem**: Initial `with_capacity(RESAMPLE_CHUNK * 4)` = 4096. `cpal::BufferSize::Default` can return 4096+ samples on some CoreAudio devices; a single callback could push `resample_buf` past capacity, reallocating on the RT thread.
- **Fix**: Cap: if `resample_buf.len() > RESAMPLE_CHUNK * 16`, log warning and drain excess. Initialize with larger fixed capacity (`RESAMPLE_CHUNK * 32`).
- **Risk**: Low.

### AU-NEW-6. `mono_buf` pre-alloc claim is partially false [low]

- **Location**: `audio_engine.rs:142, 199-200`
- **Problem**: `mono_buf` capacity 8192 is correct for today's native rates but not guaranteed. `mono_buf.extend_from_slice(data)` reallocates if `data.len() / native_ch > 8192`. Violates the "zero allocations per callback" claim at `:138`.
- **Fix**: Compute safe upper bound from actual max buffer size; document the 8192 assumption explicitly.
- **Risk**: Negligible.

### AU-NEW-7. `cleanup_old_segments` and `cleanup_old_audio` each call `list_wavs` — two walks per timer tick [low]

- **Location**: `audio_dir.rs:62-76, 80-91, 109-113`
- **Problem**: `spawn_cleanup_timer` calls both functions sequentially, each doing a full `read_dir + collect + sort`. Resolves deferred AU6.
- **Fix**: `cleanup_all(audio_dir, max_segments, max_age_secs)` that calls `list_wavs` once.
- **Risk**: None.

### AU-NEW-8. `write_error_logged` flag is never reset on rotation if `open_new_segment` fails [medium]

- **Location**: `audio_engine.rs:49, 248-254, 288`
- **Problem**: `write_error_logged` is reset to `false` only in the `Ok` branch of `WavWriter::create`. If `open_new_segment` fails (see AU-NEW-4), the flag stays `true` and subsequent write-sample errors are silently swallowed — the opposite of the intent.
- **Fix**: Reset `write_error_logged = false` unconditionally at the top of `open_new_segment`.
- **Risk**: None.

### AU-NEW-9. `rotate_segment` calls `open_new_segment` before sending the completed path [low]

- **Location**: `audio_engine.rs:297-332`
- **Problem**: If `open_new_segment` panics after `send_path` is set but before `send(p)`, the completed segment path is lost. `parking_lot::Mutex` poisons on panic, killing the audio thread.

### AU-NEW-10. `find_orphan_segments` relies on filename sort order, not mtime [low]

- **Location**: `audio_dir.rs:49-58, 97-103`
- **Problem**: `list_wavs` sorts lexicographically, assumes `segment_YYYYMMDD_HHMMSS.wav` grammar. Non-canonical filenames corrupt sort order.

# Pass-3 Review: Transcriber & Python Daemon

## Summary

Ten new findings across `transcriber.rs` and `mlx_transcribe.py`. The most critical are structural: `read_line_bounded` silently consumes bytes permanently from the `BufReader` whenever a line is near (but under) the cap — a class of bug invisible in tests because `Cursor` is stateless. The `Drop` impl kills the child but never reaps it, leaving zombies. `--break-system-packages` is a hard blocker for managed-environment Python. The daemon lock is held for entire 5-30 s inference call. The `unwrap_or_else(Utc::now)` fallback is semantically wrong for dedup.

## Findings

### T1. `read_line_bounded` consumes bytes permanently — a near-cap line corrupts subsequent reads [High]

- **Location**: `transcriber.rs:190-210`
- **Problem**: `reader.take(max_bytes)` wraps the live `BufReader` by value; `read_until` drains bytes from the underlying buffer up to `max_bytes`. After the call, the `Take` is dropped — but the `BufReader` has already consumed those bytes; they are not restored. For a response that happens to be exactly `max_bytes - 1` bytes with no newline, `read_until` consumes all `max_bytes - 1` bytes, the check at `:197` evaluates **false**, no error raised. Next call reads mid-protocol. Current tests use `Cursor` (stateless), so they can't catch this.
- **Fix**: Rewrite using `reader.fill_buf()` + `reader.consume(take)` directly, without wrapping in `Take`. Example rewrite provided in source review.
- **Risk**: Medium. No production regression today (responses are kilobytes); the invariant is silently broken.

### T2. `Drop for Transcriber` kills the child but never waits — guaranteed zombie accumulation [High]

- **Location**: `transcriber.rs:127-133`
- **Problem**: On Unix, a killed process is a zombie until parent calls `waitpid`. The child PID occupies a slot for the entire session. On daemon restart (`transcriber.rs:54-57`), old zombies accumulate.
- **Fix**: Call `d.child.wait().ok()` after `d.child.kill().ok()`. Apply same pattern at the inline `*guard = None` sites (`transcriber.rs:65, 69`) via a `shutdown_daemon` helper.
- **Risk**: Low.

### T3. `--break-system-packages` in `ensure_package` hard-fails on managed environments [High]

- **Location**: `mlx_transcribe.py:13-21`
- **Problem**: Introduced pip 23.1. Fails on older pip with unknown-flag error, producing empty stdout that Rust interprets as `"Daemon crashed (empty response)"`. Will silently wipe on `brew upgrade python`. Incompatible with deferred M11 venv fix.
- **Fix**: Remove the flag. Replace `ensure_package` with fail-fast `try_import_or_warn` that logs to stderr and `sys.exit(1)`. The venv setup should handle install at build time.
- **Risk**: Medium. Changes first-run behavior; only acceptable with M11.

### T4. `find_script` logs no winning path — diagnostic silence [Medium]

- **Location**: `transcriber.rs:262-284`
- **Problem**: 6-candidate probe with no log of winner or rejections. When the daemon fails to start on a packaged `.app`, only a generic error is logged.
- **Fix**: `log::info!("Script found: {}", p.display())` on success; `log::debug!("Script probe miss: {}")` on each rejection; enumerate tried paths on final `Err`.
- **Risk**: None.

### T5. `python_version_ok` spawns up to 8 subprocesses serially on cold start [Medium]

- **Location**: `transcriber.rs:251-260, 233-247`
- **Problem**: Each spawn takes ~100-300 ms. Worst-case ~1.75 s before daemon starts. The UI is unresponsive during this window because `Transcriber::new()` is called synchronously from `pipeline::start`.
- **Fix**: `Path::exists()` short-circuit — skip spawn if binary doesn't exist. Parallelize with `std::thread::scope`. `OnceCell` cache.
- **Risk**: Low.

### T6. `transcribe` holds `daemon.lock()` for full 5-30 s MLX inference [Medium]

- **Location**: `transcriber.rs:49`, `pipeline.rs:79-81`
- **Problem**: Any future concurrent caller blocks for up to 30 s. Forecloses the deferred T2/T9 concurrency improvements.
- **Fix**: Split protocol into non-blocking `send_request` and blocking `wait_response`. Or at minimum document `// INVARIANT: lock is held for I/O duration`.
- **Risk**: Medium for the split.

### T7. `extract_timestamp_from_path` falls back to `Utc::now` — dedup false-negatives on foreign files [Medium]

- **Location**: `transcriber.rs:217-219`, `pipeline.rs:130-134`
- **Problem**: Non-canonical WAV → `Utc::now()` timestamp → never matches DB → re-transcribed on every restart with a different "now". `end_time = start_time + duration` becomes a future timestamp.
- **Fix**: Return `Option<DateTime<Utc>>` from `extract_timestamp_from_path`. Treat `None` as non-fatal skip.
- **Risk**: Low.

### T8. Python `path` parameter is not normalized [Low]

- **Location**: `mlx_transcribe.py:219-224`
- **Problem**: Symlinks can cause double-stat; non-UTF-8 paths produce confusing error messages.
- **Fix**: Rust: replace `audio_path.to_str().unwrap_or("")` with explicit error. Python: `os.path.realpath` before `exists`.
- **Risk**: None.

### T9. `serde_json::Value` response parsing is untyped [Low]

- **Location**: `transcriber.rs:91-114`
- **Problem**: Field renames in Python silently degrade to default via `.unwrap_or("")`. Coupling to wire protocol by string literal.
- **Fix**: Define `#[derive(serde::Deserialize)] #[serde(untagged)] enum DaemonResponse { Error {...}, Skipped {...}, Ok {...} }`.
- **Risk**: Low.

### T10. `is_audio_silent` reads entire WAV into memory before computing RMS [Low]

- **Location**: `mlx_transcribe.py:68-84`
- **Problem**: `readframes(getnframes())` + `struct.unpack` = ~3× file size peak allocation. Also has dead `swidth == 4` branch for a format never used.
- **Fix**: Chunked loop over `readframes(4096)` with running RMS accumulator. Remove the float32 branch.
- **Risk**: Low.
